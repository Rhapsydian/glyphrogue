# Glossary

Short definitions for terms this project uses in a specific, sometimes
non-obvious way, gathered from across `docs/design/*.md`. This is **not**
another deep-dive planning doc — like `docs/data-model.md`, it's a living
cross-cutting reference. Not exhaustive by design: add a term here when it's
been a source of confusion or drift, not preemptively. Each entry links to
the deep-dive doc that fully specifies it; treat that doc as the source of
truth if this glossary and it ever disagree.

## Plugin system

- **Plugin** — a dev-time, author-managed, statically-registered extension
  unit: `{ id, version, dependencies, register(api) }`. Two kinds exist,
  **Content** and **Service** (below). Distinct from **Mod** — see
  "Plugin vs. Mod" in `scripting-api.md`.
- **Mod** — the player-facing, runtime-toggleable equivalent of a Plugin: a
  persisted, player-toggleable enabled-mods settings slice read at boot.
  Deliberately out of scope everywhere so far — see "Open items carried
  forward" in `editor.md`.
- **Content plugin** — the default Plugin kind. Multi-instance: many
  entities/rules/generators/etc. can all be registered and coexist
  simultaneously, selected by id or component match. See "Plugin kinds:
  Content vs. Service" in `scripting-api.md`.
- **Service plugin** — single-slot, swappable Plugin kind, for a concern
  where exactly one implementation is active at a time (`memory`,
  `audioLoader`). Registered via `api.registerService(id, implementation)`,
  which merges the implementation's bound methods flat onto `api`. See
  "Plugin kinds: Content vs. Service" in `scripting-api.md`.
- **`register(api)`** — the one entry point every plugin module exports,
  called once at load time with either the real `api` or a recording
  stand-in (see "Derived manifest" below). See "Plugin module format" in
  `scripting-api.md`.
- **Dependency / version range** — a plugin's `dependencies` map (including
  on `core` itself), using npm-style `^`/`~`/exact semver ranges; resolved
  via `getOrderedIds`'s topological sort, a hard error on a cycle or a
  missing dependency. See `plugins.js` and "Versioning & compatibility" in
  `scripting-api.md`.
- **`override`** — a plugin field that must exactly name the id it's
  replacing; last-registered-wins once it matches, a hard error if it
  doesn't match anything already registered. The same mechanism swaps a
  Service plugin's active implementation. See "Conflict handling" in
  `scripting-api.md`.
- **Derived manifest / recording api** — a plugin's declared contents are
  never hand-authored. Dev tooling calls `register()` against a recording
  stand-in of `api` that logs each `registerX` call instead of mutating
  real state, and derives the manifest from that log. See "Plugin module
  format" in `scripting-api.md` and `recordingApi.js`.

## Content kinds

All id-registered via a `registerX(id, ...)` call on `api`, with
conflict/override checking:

- **Entity** — `registerEntity(id, def)`. An entity definition: inert
  component data, not behavior.
- **EntityType** — `registerEntityType(id, def)`. Sugar bundling an Entity
  definition with its own inline Rules, auto-scoped to its own instances.
- **Rule** — `registerRule(id, actionType, ruleFn)`. Behavior keyed to an
  Action type, matched by component, run as one entry in an ordered
  pipeline for that action type.
- **Generator** — `registerGenerator(id, generatorFn)`. `params + seeded
  RNG → ZoneDTO`. The four first-party generators (BSP, cellular automata,
  minimal WFC, layered biome) ship as Content plugins — see
  `mapgen-and-editor.md`.
- **Screen** — `registerScreen(id, definition)`. A UI screen definition,
  opened via `openScreen`, held open by `lock()`.
- **Sound** — `registerSound(id, definition)`.
- **ScriptedEvent** — `registerScriptedEvent(id, def)`. A multi-step
  triggered sequence, tracked on a dedicated tracking entity created lazily
  on first match.

## World model

Ad-hoc string tags used as matching keys — never registered anywhere:

- **Component** — a plain string type key passed to `addComponent(world,
  entity, type, data)`. No registration step exists for a component type;
  any string works.
- **Action** — a plain string `type` on a dispatched action object, matched
  by Rules via their declared `actionType`. No `registerAction` exists
  anywhere in the codebase — same unregistered-tag shape as Component, not
  a Content kind.
- **World** — the live ECS instance (entities + components), created by
  `createWorld()`.
- **Region** — world-tier resolution: one world-map cell, generated by the
  same generator interface a Zone uses, at a coarser scale.
- **Zone** — a local, generated-or-hand-authored playable area; the unit
  `generateZone`/`loadZone` operate on.
- **`ZoneDTO`** — the shape both a generator's output and a save slice
  share; there is no separate hand-authored format to keep in sync.

## Save

Four independently-tracked version numbers, easy to conflate:

- **`coreSchemaVersion`** — governs the save DTO shape and its migrations.
  See `core-architecture.md`.
- **`gameDataVersion`** — one slice, one migration chain, owned by the
  game. Everything registered by a statically-imported (build-time) module
  writes here, including first-party Content plugins.
- **Per-plugin data version** — `plugins: { [pluginId]: { pluginDataVersion,
  payload } }`, independently versioned per dynamically-loaded plugin.
- **`CORE_API_VERSION`** — independent of the save schema; what a plugin's
  `dependencies.core` range checks against. Bumped when `register()`'s
  surface changes, not when the save DTO does. See `plugins.js` and
  "Versioning & compatibility" in `scripting-api.md`.
