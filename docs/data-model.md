# Data model reference

A living reference for `packages/core`'s actual data shapes — components,
action payloads, save DTOs, and content/asset formats. This is **not**
another `docs/design/*.md` planning-session doc; those are point-in-time
decision records (why a choice was made). This doc tracks the *current,
real* shape of things as implementation defines them.

**Maintenance rule**: update this doc as part of the implementation work
that defines or changes a shape — not speculatively ahead of code. Every
entry below is currently an **illustrative example** sketched in a design
doc, not a committed schema; as real code lands, replace the design-doc
cross-reference with a pointer at the actual source file/type and drop the
"illustrative" tag. A stale entry here is worse than no entry — if
something changes and this doc isn't updated in the same PR, that's a bug
in the PR, not a documentation backlog item.

| `Health` | `{current, max}` | `core-architecture.md` | illustrative |
| `Inventory` | `{items}` | `core-architecture.md` | illustrative |
| `ExplodesOnDeath` | marker (no data) | `scripting-api.md` | illustrative |
| `Wanders` / `ChasesPlayer` / `Flees` / `Guards` | markers, first-party `TakeTurn` behaviors — `Wanders` optionally carries `{ lastDirection }` (cycling state), `ChasesPlayer`/`Flees` optionally carry `{ radius }` (FOV range, defaults to 8), `Guards` carries `{ post: {x,y} }` | `packages/core/src/behaviors.js` | implemented |

| `ScriptedEvent` | `{ id }` — marker tagging a scripted event's tracking entity so `registerScriptedEvent`'s compiled rules can find it via `ctx.query` (not a JS-side id→entity map, so this state saves/loads for free) | `packages/core/src/scriptedEvents.js` | implemented |
| `Timer` | `{ action }` — a one-shot scheduler actor (added with a negative initial budget) that dispatches its carried action once `engine.js`'s `act()` picks it, then self-removes; used by `registerScriptedEvent`'s `timeUnits` waitFor | `packages/core/src/engine.js` | implemented |
| `EntityType` | `{ type }` — tags every entity `instantiateEntity` creates with the definition id it came from, so `registerEntityType`'s inline rules can auto-scope to their own instances | `packages/core/src/definitions.js` | implemented |
| `PendingUI` | `{ screenId, payload }` — generic core→UI hand-off marker; supersedes the earlier `PendingDialogue`-only concept (`ui-and-input.md`'s `ShowDialogue` is now `PendingUI`'s first built-in consumer, though `ShowDialogue`/scripted events themselves are still session 25's job) | `packages/core/src/api.js` (`openScreen`/`closeScreen`) | implemented |
| `PlayerControlled` | marker (no data) — distinguishes the player-controlled actor so `engine.js`'s `act()` locks and waits for external input instead of auto-dispatching `TakeTurn` | `packages/core/src/engine.js` | implemented |
| `Memory` | `{ remembered: Set<string>, lastKnownLight: Map<string, {level,color}> }` — optional first-party convenience component wiring `visibility.js`'s pure remembered-cells/last-known-light functions onto an entity via `memory.js`'s `ensureMemory`/`updateEntityMemory`; not core-mandated, same "first-party but not mandatory" precedent as `Wanders`/`ChasesPlayer`/`Flees`/`Guards` — a game may manage exploration memory itself instead | `packages/core/src/memory.js` | implemented |

## Save DTO shapes

Registry state (registered rules/generators/etc.) is deliberately **not**
part of the save DTO below — it's code, reconstructed by re-running a
game's own `register*` calls at boot, same as any other content. Only
world/scheduler/rng *state* is persisted.

| Shape | Notes | Source | Status |
|---|---|---|---|
| Top-level save | `{ coreSchemaVersion, core, gameDataVersion, game, plugins: { [pluginId]: { pluginDataVersion, payload } } }` — `core` is `{ nextId, entities, components, scheduler: { roundBudget, actors }, rng: { state } }`. `game` is produced by an injected `serializeGame` hook. `plugins` is populated by `serialize()`'s `plugins` list of loaded-plugin descriptors (`{ id, pluginDataVersion, serialize(api) }`, from `plugins.js`'s `loadPlugins()`), one independently-versioned slice per plugin; `deserialize()`'s own `plugins` list (`{ id, deserialize(slice, api) }`) receives each slice back on load. A save slice for a plugin not currently installed is a hard load-time error (`checkPluginsPresent`) | `packages/core/src/save.js` | implemented |
| Migration mechanism | `runMigrations(payload, fromVersion, toVersion, migrations)` — sparse stepwise chain keyed by target version. `core`'s own migrations are run by `deserialize()` directly; `game`/plugin slices don't get an automatic migration step from core — their own `serializeGame`/`deserializeGame`/per-plugin `deserialize` hooks call `runMigrations` themselves if they need to, using the `gameDataVersion`/`pluginDataVersion` they receive | `packages/core/src/save.js` | implemented |
| Storage backend | `{ save(key, data): Promise<void>, load(key): Promise<data\|undefined> }` — `createMemoryStorage`/`createLocalStorageBackend`/`createFsStorage` (atomic temp-file-then-rename), selected per build target | `packages/core/src/storage.js` | implemented |
| `ZoneDTO` | `{ width, height, cells: string[] (row-major), entities: [{type,x,y,data}] (blueprint placements, not live ECS entities), anchors: [{id,x,y}], logicalLinks: [{id,from:{x,y},to:{x,y}\|{zoneId,x,y},bidirectional}] }` — a template is the same shape, stamped via `stampTemplate` | `packages/core/src/mapgen.js`, `packages/core/src/zoneComposition.js` | implemented |
| Zone diff/overlay | `{ cellOverrides: [{x,y,cell}], entityDiffs: [{op:'add'\|'remove'\|'modify',...}] }` — the one mechanism serving both author overrides and player-mutation saves; applied via `applyDiff` | `packages/core/src/zoneDiff.js` | implemented — diff carrying mod-defined entity types still open, see `BACKLOG.md` |
| Settings slice | keybindings, volume/mixing — persists outside save data, own storage key | `ui-and-input.md`, `audio.md` | implemented (`packages/input/src/keybindingStorage.js`, `packages/core/src/audioSettings.js`) |

## Content/asset formats

| Format | Notes | Source | Status |
|---|---|---|---|
| Tileset entry | `symbol -> { fontFace, codepoint, foreground, background }` — `codepoint` is a uniform lowercase hex string across every font source (generalizing Pixelyph's private-use-area convention to standard-font symbols too); `foreground`/`background` stay unresolved (raw or `{ token }`) until draw time | `packages/core/src/tileset.js` | implemented |
| Palette/theme object | `{ tokens: { [id]: string \| GradientDescriptor } }` — token vocabulary + raw-color escape hatch, one object both render paths read (DOM path itself still unbuilt, no UI framework exists yet); a `{ token }` wrapper resolves one level (no chained indirection) | `packages/core/src/palette.js` | implemented |
| Gradient descriptor | `{ type: 'gradient', stops: [{offset, color}], direction: 'vertical' \| 'horizontal' }` — `direction` defaults to `'vertical'`; a stop's `color` may itself be a `{ token }` reference | `packages/core/src/palette.js` | implemented |
| Font-source calibration record | `{ scale, baselineOffset, horizontalCenteringMode }` — default derived from metrics (`unitsPerEm`/`ascender`/`descender`, not raster measurement) against a pinnable reference ratio, override-able and persisted via `registry.js`'s `options.override` | `packages/core/src/fontSources.js` | implemented |
| Pixelyph glyph manifest | `{ meta, glyphs }`, `glyphs` keyed by lowercase hex codepoint | `fonts-and-tilesets.md`, consumed by `packages/core/src/pixelyphImport.js` | implemented on both sides |

## Registration & API surface

| Call/shape | Notes | Source | Status |
|---|---|---|---|
| `createApi({ roundBudget, seed, platform, isWalkable?, isOpaque? })` | the one public inspection/mutation surface every consumer goes through — bound methods over one internal world+registry+scheduler+engine instance, no manual world/registry threading. `isWalkable`/`isOpaque` are the caller-injected map query `findPath`/`computeFov` need (core still owns no grid/zone storage); same DI shape as `platform`/`storage`/`rng`, both optional | `packages/core/src/api.js` | implemented |
| `api.register*(id, def, options?)` | `registerRule`/`registerGenerator`/`registerScreen`/`registerSound`/`registerEntity`/`registerEntityType`/`registerScriptedEvent` are all implemented via `createApi` — the full registration surface `scripting-api.md` describes | `scripting-api.md` and each feature doc | implemented |
| `registerRule(registry, id, actionType, ruleFn, { priority?, components?, reads?, writes? })` | `components` is a declarative entity-gating filter (`{ all?, any?, none? }`, each an array of a bare component-type string (presence-only) or `{ component, ...operators }` (presence + partial-match field comparisons via `equals`/`notEquals`/`gt`/`gte`/`lt`/`lte`/`in`/`notIn`)), enforced by `dispatch()`/`dispatchExclusive()` before `ruleFn` runs. `reads`/`writes` are optional declared component-type lists, made load-bearing only under an opt-in dev-mode `ctx` wrapper (a `devMode` flag threaded through `dispatch`/`dispatchExclusive`/`createEngine`/`createApi`) that throws on any undeclared `ctx.getComponent`/`addComponent` access; zero cost when `devMode` is off | `packages/core/src/actions.js` | implemented |
| `registerGenerator(registry, id, generatorFn, { paramsDefaults? })` | `paramsDefaults` is a load-bearing values-only object (no schema/types); `generateZone` merges it into `params` whenever a field is omitted, caller-supplied values winning | `packages/core/src/mapgen.js` | implemented |
| `registerEntity(registry, id, { components }, options?)` / `getEntityDefinition` / `instantiateEntity(registry, world, id, overrides?)` | inert component-bag definitions; `instantiateEntity` creates a live entity from a registered definition, shallow-cloning each component's data per instance and tagging the entity with `EntityType: { type: id }` | `packages/core/src/definitions.js` | implemented |
| `registerEntityType(registry, id, { components, rules }, options?)` | sugar decomposing into one `registerEntity` call plus N `registerRule` calls, each auto-scoped to this type's own instances via a `components: { all: [{ component: 'EntityType', equals: { type: id } }] }` filter (`registerRule`'s components-filter mechanism, not a hand-rolled guard) | `packages/core/src/definitions.js` | implemented |
| `registerScriptedEvent(registry, id, { trigger, steps }, options?)` / `getScriptedEvent` | compiles a step list into a trigger rule plus one rule per `waitFor` step; `waitFor: { action, ...fields, predicate? }` matches field-equality (ANDed with an optional predicate) against a dispatched action, `waitFor: { timeUnits }` schedules a `Timer` entity instead. Progress lives in `EventState` on a dedicated tracking entity (`ScriptedEvent` marker) created lazily on first trigger match; re-triggering an in-progress event is a no-op, a completed (tracking entity destroyed) event can start fresh on a later trigger match | `packages/core/src/scriptedEvents.js` | implemented |
| `loadPlugins(api, plugins, { coreApiVersion? })` / `satisfiesRange(version, range)` / `CORE_API_VERSION` | a plugin is `{ id, version, dependencies, register(api) }`; load order reuses `registry.js`'s generic `getOrderedIds` topological sort applied to plugin ids (`core` excluded from the graph, checked separately). `satisfiesRange` is a hand-rolled `^`/`~`/exact comparator against `major.minor.patch` only (no pre-release tags or range combinators) — this project's first version-range check, deliberately narrower than full semver to stay at zero runtime dependencies. Missing dependency, dependency cycle, and incompatible version (core or plugin-to-plugin) are all hard load-time errors | `packages/core/src/plugins.js` | implemented |
| `createRecordingApi()` | returns `{ api, manifest }` — `api` mirrors only the `register*` surface, logging each call into `manifest` (a flat list, `{ kind, id, ...summary }`, in call order) instead of mutating any real world/registry state. The doc's "manifest is derived, never hand-authored" mechanism: a plugin's `register(api)` run against this instead of a real api produces its manifest for editor content-browsing or load-time validation | `packages/core/src/recordingApi.js` | implemented |
| `registerScreen(registry, id, { render, onOpen, onClose }, options?)` / `getScreen(registry, id)` | thin `registry.js` wrappers, same shape as `registerGenerator`; `api.openScreen(entity, screenId, payload)` sets `PendingUI` + `lock()`s, `api.closeScreen(entity, action)` clears `PendingUI` and delegates to `resolvePlayerAction` (dispatch + spend cost + unlock) — "closing is no different in kind from the engine picking back up after any other locked turn" | `packages/core/src/screen.js`, `packages/core/src/api.js` | implemented |
| `registerSound(registry, id, { trigger, source, match? }, options?)` | reactive core→audio hand-off (`audio.md`) — `dispatch()`/`dispatchExclusive()` (`actions.js`) automatically enqueue a `{ kind: 'sound', soundId, source }` render event for every resolved action whose type matches a registered sound's `trigger` (and optional `match` predicate), not an opt-in step a consumer calls separately | `packages/core/src/sound.js`, `packages/core/src/actions.js` | implemented |
| `playSound(audioCtx, buffer, { volume? })` / `playMusic(audioCtx, buffer, { volume?, loop? })` | the only module touching real `AudioContext`/`AudioBufferSourceNode`/`GainNode` calls, mirroring `glyphRenderer.js`'s posture; `buffer` is an already-decoded `AudioBuffer`-like object — decoding is the caller's concern, same "already-loaded data in" boundary as font/tileset manifests. No swappable backend, per `audio.md` | `packages/core/src/audio.js` | implemented |
| `createAudioLoader()` / `loadBuffer(loader, audioCtx, id, arrayBuffer)` / `getBuffer(loader, id)` | optional first-party convenience wrapping `decodeAudioData` + a cache keyed by id — same "first-party but not mandatory" precedent as `memory.js`; takes an already-fetched `ArrayBuffer`, never performs `fetch` itself (`core` has no network-I/O module anywhere; fetching the bytes stays a one-line caller concern) | `packages/core/src/audioLoader.js` | implemented |
| `saveMixSettings(storage, settings, key?)` / `loadMixSettings(storage, defaults, key?)` | volume mixing (`{ master, music, sfx }`, plain 0-1 multipliers) persisted as its own settings slice outside save data, reusing `storage.js`'s existing `{save, load}` backends directly — same shape `packages/input/src/keybindingStorage.js` established, no wrapper accessors needed since volumes carry no reverse-lookup/rebind logic the way keybindings do | `packages/core/src/audioSettings.js` | implemented |
| Entity/component methods | `createEntity`, `destroyEntity`, `addComponent`, `removeComponent`, `getComponent`, `hasComponent`, `query`, `getComponentsForEntity`, `dispatch`, `findPath`, `computeFov` — bound on `createApi()`, same shape `ctx` uses inside rules | `packages/core/src/api.js` | implemented |
| `dispatch(world, registry, action, mapQuery?, renderEvents?, scheduler?)` / `dispatchExclusive(...)` | `scheduler` is a further optional trailing param (same precedent as `renderEvents`) so `ctx.addActor`/`ctx.removeActor` can schedule/remove a scheduler actor from inside a rule (added for `registerScriptedEvent`'s `timeUnits` waitFor); omitted, both are no-ops | `packages/core/src/actions.js` | implemented |
| `ctx` query/mutation methods | `hasComponent`, `getComponent`, `addComponent`, `removeComponent`, `createEntity`, `destroyEntity`, `query`, `getComponentsForEntity`, `findPath`, `computeFov`, `enqueueRenderEvent`, `addActor`, `removeActor` — `findPath`/`computeFov` close over the `isWalkable`/`isOpaque` injected at `createApi()`; `enqueueRenderEvent`/`addActor`/`removeActor` are no-ops if their queue/scheduler wasn't supplied to `dispatch`/`dispatchExclusive` | `packages/core/src/actions.js` | implemented |
| `findPath(from, to, { isWalkable, maxNodes? })` | A* over 4-directional adjacency, Manhattan heuristic; returns ordered steps from `from` to `to` (exclusive of `from`), `[]` if already there, `null` if unreachable. `isWalkable` must reject out-of-bounds cells or an unreachable goal becomes an unbounded search | `packages/core/src/pathfinding.js` | implemented |
| `computeFov(origin, radius, { isOpaque })` / `fovContains(fov, x, y)` | recursive shadowcasting (8-octant); returns the `Set` of `"x,y"`-keyed visible cells (origin always included, an opaque cell is itself visible but blocks what's behind it). Shared by player FOV (`visibility.js`, session 21), per-monster perception (`behaviors.js`), and light propagation (`visibility.js`'s `computeLighting`, session 21) - one primitive, three consumers, as designed | `packages/core/src/fov.js` | implemented |
| `platform` | `{ unlockAchievement(id) }`, no-op default, injectable at `createApi()` creation — same dependency-injection shape as `storage` | `packaging.md`, `packages/core/src/api.js` | implemented |
| `rng` | seeded PRNG (mulberry32), `rng.state` a plain number so save/load can snapshot/restore it directly | `packages/core/src/rng.js` | implemented |
| `GenerationContext` | `{ rng, params, getNeighborZone }` — `rng` is seeded deterministically from `(worldSeed, zoneId)` (FNV-1a hash into `rng.js`'s `createRng`); `getNeighborZone` is a caller-injected lookup (not core-owned zone storage — zone id stays an opaque caller-defined key, no grid/coordinate system built in `core`) | `packages/core/src/mapgen.js` | implemented |
| `registerGenerator(id, generatorFn)` / `generateZone(registry, args)` / `loadZone(registry, args)` | `generateZone` builds the `GenerationContext` and calls the registered generator; `loadZone` = `generateZone` + `applyDiff`, the seed+diff save-strategy round trip; both bound on `createApi()`, defaulting `worldSeed` to the api's own `seed` | `packages/core/src/mapgen.js`, `packages/core/src/zoneDiff.js` | implemented |
| `stampTemplate` / `carveCellularAutomata` / `connectCorridor` / `runConnectivityPass` | composition primitives generator functions call into; `runConnectivityPass` is the mandatory post-generation reachability pass (physical adjacency + `logicalLinks` edges), auto-connecting unreached stamps and respecting `mayBeIsolated`/`reachableVia` | `packages/core/src/zoneComposition.js` | implemented |
