# Data model reference

A living reference for `packages/core`'s actual data shapes — components,
action payloads, save DTOs, and content/asset formats. This is **not**
another `docs/design/*.md` planning-session doc; those are point-in-time
decision records (why a choice was made). This doc tracks the *current,
real* shape of things as implementation defines them.

**Maintenance rule**: update this doc as part of the implementation work
that defines or changes a shape — not speculatively ahead of code. Every
entry below is currently an **illustrative example** sketched in a design
doc, not a committed schema; as real code lands, replace the design-doc
cross-reference with a pointer at the actual source file/type and drop the
"illustrative" tag. A stale entry here is worse than no entry — if
something changes and this doc isn't updated in the same PR, that's a bug
in the PR, not a documentation backlog item.

| `Health` | `{current, max}` | `core-architecture.md` | illustrative |

| `EventState` | `{ step }` — multi-step scripted-event progress | `scripting-api.md` | illustrative |

| `PlayerControlled` | marker (no data) — distinguishes the player-controlled actor so `engine.js`'s `act()` locks and waits for external input instead of auto-dispatching `TakeTurn` | `packages/core/src/engine.js` | implemented |
| `Memory` | `{ remembered: Set<string>, lastKnownLight: Map<string, {level,color}> }` — optional first-party convenience component wiring `visibility.js`'s pure remembered-cells/last-known-light functions onto an entity via `memory.js`'s `ensureMemory`/`updateEntityMemory`; not core-mandated, same "first-party but not mandatory" precedent as `Wanders`/`ChasesPlayer`/`Flees`/`Guards` — a game may manage exploration memory itself instead | `packages/core/src/memory.js` | implemented |

## Action types

| Type | Notes | Source | Status |
|---|---|---|---|
| `Move`, `Attack`, `Damage`, `Death` | core's first-party default combat/movement vocabulary — swappable per-encounter, not exclusive (`custom-ui-and-interactions.md`); the four first-party `TakeTurn` behaviors emit `{ type: 'Move', entity, to: {x,y}, cost }` follow-ons, but applying a `Move` (mutating `Position`) is still game content - no session has shipped a `Move`-applying rule | `core-architecture.md` | illustrative |
| `OpenDoor` | example non-combat action | `core-architecture.md` | illustrative |
| `TakeTurn` | core-shipped, dispatched per non-player actor's turn via `dispatchExclusive` (mutually-exclusive winner-take-all resolution, not the additive pipeline other action types use); zero time-units cost itself | `packages/core/src/engine.js`, `packages/core/src/behaviors.js` | implemented |
| `EnterRegion` | scripted-event trigger example (map-region entry) | `scripting-api.md` | illustrative |
| `EventTimerElapsed` | synthetic action emitted by a scheduled timer entity for `waitFor: { timeUnits }` | `scripting-api.md` | illustrative |
| Author-defined closing actions (`ResolveSkillCheck`, `ResolveBattle`, ...) | naming is author's choice, not core-mandated | `custom-ui-and-interactions.md` | illustrative |
| `cost` field (optional, on any action) | variable per-instance time-units cost (`{ type: 'Move', entity, cost: 100 }`) — `engine.js`'s `act()`/`resolvePlayerAction()` sum `cost` across every action a turn resolves to and spend the total from the scheduler | `packages/core/src/engine.js` | implemented |

## Save DTO shapes

Registry state (registered rules/generators/etc.) is deliberately **not**
part of the save DTO below — it's code, reconstructed by re-running a
game's own `register*` calls at boot, same as any other content. Only
world/scheduler/rng *state* is persisted.

| Shape | Notes | Source | Status |
|---|---|---|---|
| Top-level save | `{ coreSchemaVersion, core, gameDataVersion, game, mods: { [modId]: { modDataVersion, payload } } }` — `core` is `{ nextId, entities, components, scheduler: { roundBudget, actors }, rng: { state } }`; `game`/`mods` are opaque payloads produced by an injected `serializeGame` hook / passed through as-is (nothing produces real mod slices yet — no mod-loading system exists) | `packages/core/src/save.js` | implemented |
| Migration mechanism | `runMigrations(payload, fromVersion, toVersion, migrations)` — sparse stepwise chain keyed by target version, reused by the core slice now and available for game/mod slices once those exist | `packages/core/src/save.js` | implemented |
| Storage backend | `{ save(key, data): Promise<void>, load(key): Promise<data\|undefined> }` — `createMemoryStorage`/`createLocalStorageBackend`/`createFsStorage` (atomic temp-file-then-rename), selected per build target | `packages/core/src/storage.js` | implemented |
| `ZoneDTO` | `{ width, height, cells: string[] (row-major), entities: [{type,x,y,data}] (blueprint placements, not live ECS entities), anchors: [{id,x,y}], logicalLinks: [{id,from:{x,y},to:{x,y}\|{zoneId,x,y},bidirectional}] }` — a template is the same shape, stamped via `stampTemplate` | `packages/core/src/mapgen.js`, `packages/core/src/zoneComposition.js` | implemented |
| Zone diff/overlay | `{ cellOverrides: [{x,y,cell}], entityDiffs: [{op:'add'\|'remove'\|'modify',...}] }` — the one mechanism serving both author overrides and player-mutation saves; applied via `applyDiff` | `packages/core/src/zoneDiff.js` | implemented — diff carrying mod-defined entity types still open, see `BACKLOG.md` |
| Settings slice | keybindings, volume/mixing — persists outside save data, own storage key | `ui-and-input.md`, `audio.md` | illustrative |

## Content/asset formats

| Format | Notes | Source | Status |
|---|---|---|---|
| Tileset entry | `symbol -> { fontFace, codepoint, foreground, background }` — `codepoint` is a uniform lowercase hex string across every font source (generalizing Pixelyph's private-use-area convention to standard-font symbols too); `foreground`/`background` stay unresolved (raw or `{ token }`) until draw time | `packages/core/src/tileset.js` | implemented |
| Palette/theme object | `{ tokens: { [id]: string \| GradientDescriptor } }` — token vocabulary + raw-color escape hatch, one object both render paths read (DOM path itself still unbuilt, no UI framework exists yet); a `{ token }` wrapper resolves one level (no chained indirection) | `packages/core/src/palette.js` | implemented |
| Gradient descriptor | `{ type: 'gradient', stops: [{offset, color}], direction: 'vertical' \| 'horizontal' }` — `direction` defaults to `'vertical'`; a stop's `color` may itself be a `{ token }` reference | `packages/core/src/palette.js` | implemented |
| Font-source calibration record | `{ scale, baselineOffset, horizontalCenteringMode }` — default derived from metrics (`unitsPerEm`/`ascender`/`descender`, not raster measurement) against a pinnable reference ratio, override-able and persisted via `registry.js`'s `options.override` | `packages/core/src/fontSources.js` | implemented |
| Pixelyph glyph manifest | `{ meta, glyphs }`, `glyphs` keyed by lowercase hex codepoint | `fonts-and-tilesets.md`, consumed by `packages/core/src/pixelyphImport.js` | implemented on both sides |

## Registration & API surface

| Call/shape | Notes | Source | Status |
|---|---|---|---|
| `createApi({ roundBudget, seed, platform, isWalkable?, isOpaque? })` | the one public inspection/mutation surface every consumer goes through — bound methods over one internal world+registry+scheduler+engine instance, no manual world/registry threading. `isWalkable`/`isOpaque` are the caller-injected map query `findPath`/`computeFov` need (core still owns no grid/zone storage); same DI shape as `platform`/`storage`/`rng`, both optional | `packages/core/src/api.js` | implemented |
| `api.register*(id, def, options?)` | `registerRule`/`registerGenerator`/`registerScreen` are implemented via `createApi`; `registerEntity`, `registerEntityType`, `registerScriptedEvent`, `registerSound` don't exist yet — later sessions | `scripting-api.md` and each feature doc | partially implemented |
| `registerScreen(registry, id, { render, onOpen, onClose }, options?)` / `getScreen(registry, id)` | thin `registry.js` wrappers, same shape as `registerGenerator`; `api.openScreen(entity, screenId, payload)` sets `PendingUI` + `lock()`s, `api.closeScreen(entity, action)` clears `PendingUI` and delegates to `resolvePlayerAction` (dispatch + spend cost + unlock) — "closing is no different in kind from the engine picking back up after any other locked turn" | `packages/core/src/screen.js`, `packages/core/src/api.js` | implemented |
| Entity/component methods | `createEntity`, `destroyEntity`, `addComponent`, `removeComponent`, `getComponent`, `hasComponent`, `query`, `dispatch`, `findPath`, `computeFov` — bound on `createApi()`, same shape `ctx` uses inside rules | `packages/core/src/api.js` | implemented |
| `ctx` query/mutation methods | `hasComponent`, `getComponent`, `addComponent`, `removeComponent`, `createEntity`, `destroyEntity`, `query`, `findPath`, `computeFov` — the last two close over the `isWalkable`/`isOpaque` injected at `createApi()` | `packages/core/src/actions.js` | implemented |
| `findPath(from, to, { isWalkable, maxNodes? })` | A* over 4-directional adjacency, Manhattan heuristic; returns ordered steps from `from` to `to` (exclusive of `from`), `[]` if already there, `null` if unreachable. `isWalkable` must reject out-of-bounds cells or an unreachable goal becomes an unbounded search | `packages/core/src/pathfinding.js` | implemented |
| `computeFov(origin, radius, { isOpaque })` / `fovContains(fov, x, y)` | recursive shadowcasting (8-octant); returns the `Set` of `"x,y"`-keyed visible cells (origin always included, an opaque cell is itself visible but blocks what's behind it). Shared by player FOV (`visibility.js`, session 21), per-monster perception (`behaviors.js`), and light propagation (`visibility.js`'s `computeLighting`, session 21) - one primitive, three consumers, as designed | `packages/core/src/fov.js` | implemented |
| `platform` | `{ unlockAchievement(id) }`, no-op default, injectable at `createApi()` creation — same dependency-injection shape as `storage` | `packaging.md`, `packages/core/src/api.js` | implemented |
| `rng` | seeded PRNG (mulberry32), `rng.state` a plain number so save/load can snapshot/restore it directly | `packages/core/src/rng.js` | implemented |
| `GenerationContext` | `{ rng, params, getNeighborZone }` — `rng` is seeded deterministically from `(worldSeed, zoneId)` (FNV-1a hash into `rng.js`'s `createRng`); `getNeighborZone` is a caller-injected lookup (not core-owned zone storage — zone id stays an opaque caller-defined key, no grid/coordinate system built in `core`) | `packages/core/src/mapgen.js` | implemented |
| `registerGenerator(id, generatorFn)` / `generateZone(registry, args)` / `loadZone(registry, args)` | `generateZone` builds the `GenerationContext` and calls the registered generator; `loadZone` = `generateZone` + `applyDiff`, the seed+diff save-strategy round trip; both bound on `createApi()`, defaulting `worldSeed` to the api's own `seed` | `packages/core/src/mapgen.js`, `packages/core/src/zoneDiff.js` | implemented |
| `stampTemplate` / `carveCellularAutomata` / `connectCorridor` / `runConnectivityPass` | composition primitives generator functions call into; `runConnectivityPass` is the mandatory post-generation reachability pass (physical adjacency + `logicalLinks` edges), auto-connecting unreached stamps and respecting `mayBeIsolated`/`reachableVia` | `packages/core/src/zoneComposition.js` | implemented |
| `ensureTraversable(zone, { entryPoints, stamps?, mode? })` | composable follow-up to `runConnectivityPass` for whatever it doesn't cover — disconnected walkable regions the base algorithm itself produced, not just unreached stamps. `mode: 'prune'` (default) converts them to `'wall'`; `mode: 'connect'` carves a corridor into each instead, preserving the content | `packages/core/src/zoneComposition.js` | implemented |
| `carveBsp(zone, rng, { region?, minPartitionSize?, roomMargin? })` / `bspGenerator` | recursive split-and-carve, region-scoped; connects sibling rooms on the way back up the recursion (self-connecting by construction, no `ensureTraversable` needed). Returns `{ rooms, entryPoint }` | `packages/core/src/bsp.js` | implemented |
| `cellularAutomataGenerator` | thin whole-zone wrapper over `carveCellularAutomata`; prunes disconnected floor pockets via `ensureTraversable({ mode: 'prune' })` by default (`pruneUnreachable: false` opts out) | `packages/core/src/cellularAutomataGenerator.js` | implemented |
| `collapseWfc(zone, rng, { region?, tiles, adjacency, maxRetries? })` / `wfcGenerator` | minimal WFC — author-declared tiles + directed per-direction adjacency rules (no implicit compatibility default), lowest-entropy collapse with forward-checking propagation, contradiction retries from the same seeded rng stream. Region-scoped. `wfcGenerator` uses `ensureTraversable({ mode: 'connect' })` to preserve collapsed content | `packages/core/src/waveFunctionCollapse.js` | implemented — a fuller sample-based WFC (pattern extraction, frequency weighting, backtracking) is backlogged, not built |
| `partitionBiomes(zone, rng, { region?, biomes, seedCount? })` / `layeredBiomeGenerator` | nearest-seed-point (Voronoi-like) base biome-region layer, region-scoped; `layeredBiomeGenerator` applies `params.stamps` as the feature layer, then `ensureTraversable({ mode: 'connect' })` | `packages/core/src/layeredBiome.js` | implemented |
| `params.stamps` on all four built-in generators | `{ template, x, y, rotation?, mayBeIsolated?, reachableVia? }[]`, applied via `stampTemplate` after the generator's own base algorithm runs, uniformly across BSP/CA/WFC/layered-biome | `packages/core/src/bsp.js`, `cellularAutomataGenerator.js`, `waveFunctionCollapse.js`, `layeredBiome.js` | implemented |
| `createGlyphMetrics({ pixelsPerEm, unitsPerEm?, baselineRow?, horizontalPadding?, fallbackAdvanceWidth?, glyphs? })` / `cellSize` / `glyphAdvance` / `baselineOffset` / `fontSizePx` | the shared glyph-metrics contract both canvas and a future DOM path derive cell size from; pure data, no `ctx`/DOM dependency - DOM CSS-custom-property wiring itself isn't built yet (no UI framework exists) | `packages/core/src/glyphMetrics.js` | implemented |
| `setLayerFont(ctx, metrics, fontFamily)` / `drawGlyphCell(ctx, metrics, cellSize, { col, row, text, color, offsetX?, baselineOffsetPx?, palette? })` / `drawCellBackground(ctx, cellSize, { col, row }, value, palette)` / `drawTileCell(ctx, metrics, cellSize, palette, command)` | the only functions touching `ctx.font`/`ctx.fillStyle`/`ctx.fillText`/`ctx.fillRect`/`ctx.createLinearGradient`. `color`/background `value` resolve through an optional `palette` (a `{ token }` or gradient descriptor); a resolved gradient becomes a real `CanvasGradient` sized to the cell. `offsetX`/`baselineOffsetPx` are optional pre-computed calibration overrides for a non-base font-source glyph. `drawCellBackground`/`drawGlyphCell` stay separate primitives (a background changes far less often than the glyph over it — see `BACKLOG.md`'s redraw-cadence-decoupling deferred item); `drawTileCell` is a convenience composing both, mirroring `paintLayer`'s role | `packages/core/src/glyphRenderer.js` | implemented |
| `createPalette(tokens)` / `resolveColor(palette, value)` | curated token palette with a raw-color escape hatch; a `{ token }` reference resolves one level against `palette.tokens`, anything else (including a raw gradient descriptor) passes through unchanged; a gradient descriptor's own stop colors may nest one `{ token }` reference each | `packages/core/src/palette.js` | implemented |
| `createFontSourceRegistry({ reference? })` / `registerFontSource(registry, id, sourceMetrics, options?)` / `getFontSource` / `deriveCalibration(sourceMetrics, referenceRatio, referenceAscenderRatio)` / `calibratedGlyphAdvance(metrics, sourceMetrics, calibration, codepoint)` / `calibratedBaselineOffset(metrics, calibration)` | multi-font-source calibration built on `registry.js` (so `options.override` already lets a game/future editor replace a stored calibration and have it persist). Default calibration is metrics-based (`unitsPerEm`/`ascender`/`descender`), not raster-measurement-based. The calibration reference is either the first font source registered, or pinned explicitly via `{ reference: sourceMetrics }` at registry-creation time — independent of registration order, so e.g. a Pixelyph icon font can be declared the intended standard regardless of when a monospace fallback gets registered | `packages/core/src/fontSources.js` | implemented |
| `createTileset()` / `registerSymbol(tileset, symbol, { fontFace, codepoint, foreground, background }, options?)` / `resolveSymbol(tileset, fontSourceRegistry, metrics, symbol)` | the tileset definition format; `resolveSymbol` converts a symbol's hex `codepoint` to its drawable character and pulls its calibrated draw-position offsets from its `fontFace`'s registered font source, but leaves `foreground`/`background` unresolved (a symbol's tokens are defaults, not exclusive, per the design doc) | `packages/core/src/tileset.js` | implemented |
| `glyphManifestToFontSource(manifest)` | pure transform from Pixelyph's shipped `{ meta, glyphs }` export shape to `fontSources.js`'s `sourceMetrics` shape, ready for `registerFontSource`; no file I/O, same DI posture as `storage.js` | `packages/core/src/pixelyphImport.js` | implemented |
| `createCamera({ x?, y?, viewportWidth, viewportHeight })` / `updateCamera(camera, subjectPosition, { deadzone?, mapWidth?, mapHeight? })` | deadzone+snap scrolling - no-op while the subject stays within the margin, otherwise shifts by exactly the overshoot then clamps to map bounds; camera state lives in the rendering layer, not core save/simulation state (`rendering.md`) | `packages/core/src/camera.js` | implemented |
| `worldToScreen` / `screenToWorld` / `screenToCanvasPixel` / `worldToCanvasPixel` / `isInViewport` | the world-grid → screen-cell → canvas-pixel coordinate pipeline | `packages/core/src/camera.js` | implemented |
| `createRenderEventQueue()` / `enqueueRenderEvent(queue, event)` | the render-event buffer: a single sequential FIFO, `event: { kind: 'animation' \| 'sound', delay?, duration?, ...payload }` — "confirmed necessary" per the deep review since the model can race ahead of the view; `ctx.enqueueRenderEvent` (`actions.js`) and `api.enqueueRenderEvent` both push onto it | `packages/core/src/renderEvents.js` | implemented |
| `createSequencerState()` / `advanceSequencer(queue, state, now, onTrigger)` | drives the queue over time: pops the next event when idle, fires `onTrigger` once `delay` elapses, advances past the event once `duration` (counted from trigger time) elapses — recurses to pull the next event at the same `now` so a zero-duration event doesn't eat a whole tick. Bound on `createApi()` (`api.createSequencerState`/`api.advanceSequencer`), closing over the api's own queue | `packages/core/src/renderEvents.js` | implemented |
| `classifyVisibility` / `updateRemembered` / `computeLighting` / `cellRenderState` / `updateLastKnownLight` / `MEMORY_TONE` | pure FOV/lighting visualization over `computeFov` — visible/remembered/unknown classification, light-source blending (summed intensity, highest-intensity source's color), and the `preserveLastKnownLight` config toggle `rendering.md` calls for. `MEMORY_TONE` (`{ token: 'remembered' }`) is a real, resolvable palette token as of session 22 — a game supplies a `'remembered'` token in its own palette | `packages/core/src/visibility.js` | implemented |
| `ensureMemory(ctx, entity)` / `updateEntityMemory(ctx, entity, currentFov, lightMap, { preserveLastKnownLight? })` | optional convenience wiring of `visibility.js`'s pure functions onto the `Memory` component | `packages/core/src/memory.js` | implemented |
| `createAnimationState()` / `startTween` / `advanceAnimation(state, now)` / `tweenedPosition` / `addTransientEffect` / `activeEffects` | advance-by-dt tween/transient-effect bookkeeping, every function pure over an explicit `now` — no real timers touched; the actual `requestAnimationFrame` driver isn't built yet (no browser runtime package exists in this monorepo) | `packages/core/src/animation.js` | implemented |
| `createLayerState()` / `terrainLayerDirty(state, key)` / `markTerrainClean` | layered canvas redraw's terrain-layer dirty check; `key: { originX, originY, mapVersion }` is an opaque caller-supplied version token (caller bumps `mapVersion` on map mutation) — same DI shape as `isWalkable`/`isOpaque`/`getNeighborZone` | `packages/core/src/renderLayers.js` | implemented |
| `terrainDrawCommands(camera, cellQuery)` / `entityDrawCommands(camera, entities, animationState, now)` / `paintLayer(ctx, metrics, cellSize, fontFamily, commands, opts)` | builds draw-command lists (pure, testable without a `ctx`) then applies them via `glyphRenderer.js` — the terrain layer redraws the full viewport only when dirty, the entity/effects layer resolves each entity's tweened position and drops off-viewport entities every frame | `packages/core/src/renderLayers.js` | implemented |
