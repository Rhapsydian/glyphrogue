# Map generation & map editor

Deep-dive design doc for Glyphrogue's procedural map generation and the
dev-time map editor built on it. Produced in the session-3 planning pass
(see `BACKLOG.md` for the roadmap this fits into). Treat this as the source
of truth for the topics below — `DESIGN.md` will be trimmed to a short
summary linking here once the doc lands, same pattern as
`docs/design/core-architecture.md` and `docs/design/rendering.md`.

Generation and the editor are both **consumers** of `core`'s public
inspection/mutation API (established in `core-architecture.md`), not
privileged internals — no backdoor access to ECS storage or world state.
Everything below reads and writes through that same API surface the game
runtime, `editor`, and (eventually) mods use.

## Generation scale: optional two-tier world + local zones

**Decision**: generation supports an optional two-tier structure. A coarse
**world/region generator** produces per-tile metadata — biome, adjacency to
neighboring tiles, points of interest — over a low-resolution map. Each
world tile then lazily generates a detailed **local zone** the first time a
player enters it, rather than the whole world being generated up front.

This is opt-in, not required: a game can skip the world tier entirely and
work with a single local-zone generator directly (a dungeon-crawler
structure, one level at a time), or skip procgen altogether and hand-author
one map. To keep that possible, the local-zone generator's interface (see
"Generator API surface" below) must not assume a world-tile context exists
— when driven by a world tier, params are populated from that tile's
metadata; when standalone, the game supplies params directly.

Architecturally, the world tier is not a separate system bolted on top — it
reuses the *same* generator interface at a coarser scale (a "zone" at
world-tier resolution is one world-map cell, generated by a generator
function like any other). One system serves both scales rather than two
independently-maintained ones.

### Save strategy: seed + diff, not full grid

**Decision**: a zone's save state is its **seed/params plus a
player-mutation diff overlay**, not the full generated cell grid. A zone
regenerates deterministically from `(world seed, zone id)`, and any player
changes — dug walls, dropped items, killed monsters — reapply from the diff
on top of that regenerated base. This follows the same DTO/save-slice
philosophy `core-architecture.md` already established ("persist only
essential state; recompute ... instead of serializing it") — here, the
"recomputed" state is the entire zone layout, not just derived caches like
FOV. It also keeps save size bounded by *how much a player actually
changed*, not by total explored-world size, which matters once the world
tier makes "total explored" potentially unbounded.

This makes strict, permanent determinism a hard requirement, not just a
nice property — see "Determinism and seed sharing" below, and the
algorithm-versioning open item it raises.

## Runtime algorithms: pluggable generator interface

**Decision**: one common **generator interface** — `params + seeded RNG →
zone data` — with multiple algorithms as interchangeable strategies behind
it, not a single chosen algorithm:

- **BSP** (Binary Space Partitioning) — recursively splits a zone into
  sub-regions, then carves rooms and corridors from the split. Produces
  structured, room-and-corridor layouts (classic dungeons).
- **Cellular automata (CA)** — iteratively applies local neighbor-count
  rules (a cell becomes floor/wall based on how many of its neighbors are
  floor/wall) to carve organic, cave-like shapes.
- **WFC** (Wave Function Collapse) — a constraint-solving algorithm that
  fills a grid with tiles satisfying declared adjacency rules, useful for
  structured-but-varied content (e.g. biome tiles that must border
  compatible biome tiles).
- **Layered biome generation** — combines multiple passes (e.g. a base
  biome layer, then feature layers on top) to build varied terrain, in the
  vein of *Caves of Qud*'s world generation.

These are strategies behind one interface, not competing architectures — a
single zone generator can **compose more than one algorithm in sequence**
(cellular automata carves a cave's base shape, then a BSP-style pass stamps
a structured room into part of it). This composability is what "layered"
means here, not just picking one algorithm per zone.

**Confirmed**: `core` ships built-in implementations of all four as
first-party generators usable out of the box. Game and mod authors can
register **additional** generator algorithms through the same interface
(see "Generator API surface" below) — this is not a closed, core-only set.

### Determinism and seed sharing

**Confirmed design goal**: players of a downstream game should be able to
**share seeds** and get the same generated world. This sharpens determinism
from a nice property into a hard constraint: every generator must be a pure
function of `(seed, params)` with zero hidden entropy — no unseeded
`Math.random()` calls, no wall-clock reads, no iteration-order-dependent
collection behavior that could vary across JS engines or versions. The RNG
itself is seeded from `(world seed, zone coordinate/id)`, so any zone is
independently reproducible without needing to replay the whole world's
generation history.

This interacts with an open question flagged below: if generator *code*
changes between game versions (a bug fix to the CA rule set, say), does an
old shared seed still reproduce the same zone? A shared seed may need to
carry a generator-algorithm version alongside it to stay meaningful across
updates — not solved in this pass, but worth having on record early since
it shapes how seeds get surfaced to players later (session 6, UI/UX).

**Deferred**: exact algorithm internals — the specific WFC constraint-
solving approach, CA neighbor-count rule sets, BSP split heuristics — are
implementation-time detail, same pattern `rendering.md` used for deferring
the shadowcasting variant.

## Generator API surface

**Decision** (sketched at signature/shape level, not full plugin packaging
— see "Open items" for what's deferred to the scripting-api session):

```
registerGenerator(id, generatorFn)
generatorFn(ctx: GenerationContext) => ZoneDTO
```

`GenerationContext` provides:

- A seeded RNG (seeded from `(world seed, zone coordinate/id)`).
- `params` — author/mod-supplied config, plus any world-tier metadata
  (biome, adjacency) when a world tier drives this zone.
- Read access to already-generated **neighboring zones**, so a corridor
  exiting one zone's edge can align with a corridor entering the adjacent
  zone rather than the two independently guessing where the boundary
  connection should be.

The output is a **`ZoneDTO`** — deliberately the same shape a save slice
stores, so freshly-generated and loaded-from-seed-plus-diff zones feed
identical downstream code (rendering, core state) with no special-casing
for "was this generated or loaded."

**Extended by `docs/design/editor.md`**: `registerGenerator` gains an
optional load-bearing `paramsDefaults` object — `generateZone` merges
declared defaults into `params` whenever a field is omitted — motivated
by the content browser and map editor's params panel needing something to
introspect. Cross-referenced there rather than re-decided here.

`core` also exposes **composition primitives** that generator functions
call into, rather than each generator reimplementing carving/pathing logic
from scratch:

- Stamp a template at a position/rotation.
- Carve a region with cellular automata.
- Connect two points with a corridor.
- Run the connectivity pass (below).

### Connectivity pass

A gap surfaced directly in discussion: nothing in a stamp-and-generate
pipeline guarantees a stamped template ends up *reachable*. A vault could
be visually placed but topologically isolated if nothing checks.

**Decision**: after all stamping and generation for a zone completes, a
**mandatory connectivity pass** runs — a flood-fill reachability check from
the zone's designated entry point(s). Reachability is modeled as a
**graph**, not pure grid adjacency, with two kinds of edges:

- **Physical edges** — normal walkable-cell adjacency (what a flood-fill
  checks by default).
- **Logical links** — teleporter pairs, portals, one-way chutes, key-gated
  doors: declared the same way a corridor connection is, but without
  carving any cells. A teleporter is a valid path, even though it isn't
  physically adjacent to anything.

Any stamped region unreachable by either kind of edge is auto-connected via
the corridor primitive, preferring a template's **declared anchor points**
(see "Composition" below) over guessing the nearest open cell — guessing
risks punching a corridor through the middle of a wall instead of an
intended doorway.

Logical links generalize across the world/local-zone tier too: a portal
connecting two zones uses the same mechanism as one connecting two points
within a single zone, not a special case bolted on separately.

**Resolved by `scripting-api.md`**: full plugin/module packaging for
generator registration — each of the four first-party generators is a
**Content plugin** (`scripting-api.md`'s "Plugin kinds: Content vs.
Service"), `register(api)` calling `registerGenerator` for its own id. One
plugin per generator, not a bundle — a game can drop e.g. the WFC generator
without carrying its plugin. Core-bundled, shipped from `@glyphrogue/core`
itself rather than a `src/plugins/<id>/` folder; the "default-enabled"
posture and the reconciliation work to get there is tracked in
`BACKLOG.md`'s "packages/core plugin reconciliation roadmap."

## Map editor: hand-authoring, tuning, and previewing

**Decision**: the editor builds directly on `core-architecture.md`'s
already-decided **static edit mode** — no new editor mode is needed for
generation work. The editor calls the same `registerGenerator` functions
with editor-controlled params/seed, through the same inspection/mutation
API the runtime uses; it has no special generation pathway.

UI concept, sketched at a conceptual level (exact layout is an
implementation-time UI/UX concern, session 6):

```
┌─ Generator: cellular-automata-cave ──────┐┌─ Preview ──────────────┐
│ seed:      [ 4f91a2c3        ] [reroll]  ││ ##########.......##### │
│ fillPct:   [====------] 45%              ││ #....##....###....#### │
│ smoothing: [======----] 3 passes         ││ #..######......##..### │
│ ...                                      ││ #....................# │
│                                          ││ ##########.......##### │
│ [pin selection]  [regenerate]            │└────────────────────────┘
└───────────────────────────────────────────┘
```

- A **params/seed panel** bound to whatever param schema the selected
  generator declares (sliders/fields, not hardcoded per-algorithm UI).
- A **live preview** using the existing rendering layer (`rendering.md`) in
  static mode.
- **Reroll** (new random seed) and **regenerate** (rerun with current seed
  and params) actions.
- A **pin/lock** mechanic — select part of a generated zone and lock it
  while rerolling the rest. This is the interactive precursor to the
  stamps/overrides mechanism below: pinning in the editor and a runtime
  override are the same underlying diff-overlay concept, one exercised
  interactively by an author, the other captured automatically from
  gameplay.

Hand-editing cells directly (already covered generically by
`core-architecture.md`'s static edit mode: "editing map cells") produces
the same `ZoneDTO` shape a generator would output. There is one zone
format, whether it came from an algorithm, a human, or both — no separate
hand-authored format to keep in sync with the generated one.

Given the seed-sharing goal above, the editor is also the natural place for
an author to preview what a specific shared or curated seed actually
produces before shipping it (e.g. a curated "challenge seed") — a usage
note on existing functionality, not a new mechanism.

## Composition: templates, stamps, overrides

**Decision**: three composition mechanisms, all built from the primitives
above and sharing one diff/overlay concept:

- **Templates** — reusable hand-authored `ZoneDTO` fragments (a room
  shape, a set-piece encounter) that a generator stamps at a chosen
  position/rotation during generation, via the "stamp a template"
  primitive. Templates **declare connection points** — anchor cells
  marking where a corridor is meant to attach (e.g. a doorway) — so the
  connectivity pass has an explicit, sensible attachment point instead of
  guessing nearest-cell.
- **Stamps** — one-off placement of a template or a direct hand-edit,
  applied *after* algorithmic generation runs ("always put this specific
  vault here, regardless of what the cave generator produced around it").
  Stamps carry a reachability property:
  - Default: **must be reachable**, enforced by the connectivity pass.
  - `mayBeIsolated: true` — opts out, for content intentionally
    unreachable by normal means (visible treasure the player can't
    directly reach, a design pattern some roguelikes use deliberately).
  - `reachableVia: <link-id>` — points at a specific logical link (e.g. a
    teleporter placed alongside the stamp) so the connectivity pass
    confirms reachability through that link rather than flagging the stamp
    as broken for lacking a corridor.
- **Overrides** — a persistent diff layer that forces specific
  cells/entities regardless of what the generator produced, without
  forking the generator itself. This reuses the *same diff mechanism* as
  the player-mutation-diff save strategy above — an author's override and
  a player's saved changes are the same kind of object, authored at
  different times by different people.

Net authoring workflow, whether by a game author or at runtime by a player:
**generate (or start blank) → stamp templates → hand-edit remaining cells →
persist as overrides.** One mechanism serves both a game author's
guaranteed set-piece zone and a player's saved runtime mutations, rather
than two parallel systems.

## Open items carried forward

- Exact algorithm implementations (WFC constraint-solving approach, CA
  rule sets, BSP split heuristics) — implementation time.
- Full plugin/module packaging for generator registration (discovery,
  versioning of the registration mechanism, how a mod's package declares
  its generators) — scripting-api session (topic 4).
- World↔local edge-matching/connectivity algorithm specifics beyond the
  logical-link model above (exact boundary-alignment algorithm) —
  implementation time.
- Zone diff/overlay storage format specifics — likely finalized alongside
  session 4's mod-defined save-slice work, since overrides may need to
  carry mod-defined entity types.
- Algorithm/seed versioning for long-term seed-sharing reproducibility
  across game updates — flagged as a real requirement (see "Determinism
  and seed sharing" above), not solved in this pass.
