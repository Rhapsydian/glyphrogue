## Key decisions

- **`@glyphrogue/core` is a `peerDependency`, `svelte`/
  `@sveltejs/vite-plugin-svelte` are `devDependencies` only** — matches
  `editor.md`'s decision exactly: a downstream game supplies the live
  `api`/world (no dual-instance risk) and never needs the Svelte compiler
  installed. `dist/` (built via `vite build`, wired to `prepublishOnly`) is
  the only thing `files` publishes; `css: 'injected'` keeps `dist/mount.js`
  a single self-contained file instead of a second `dist/mount.css` a
  consumer would have to remember to import.
- **`hotReload.js` is deliberately hot-agnostic (no `import.meta.hot`
  inside it at all)** — Vite keeps only the *last* `hot.dispose()`
  registration per module, a single slot rather than a queue. The first
  design registered world-snapshotting and editor-unmounting as two
  separate `hot.dispose()` calls; the second silently clobbered the first,
  and restores kept coming back empty. Fixed by exporting plain
  `snapshotWorld`/`restoreWorldFromSnapshot` helpers with no `hot`
  parameter, leaving the *one* `hot.dispose()` registration to whichever
  caller needs to combine multiple teardown concerns into it — discovered
  by hand during manual HMR verification, not by test failure.
- **`writeFileAtomic` extracted from `packages/core/src/storage.js`
  rather than reusing `createFsStorage` directly** — the shared file-write
  API needs to write caller-supplied raw content to a caller-supplied
  project path; `createFsStorage`'s `save(key, data)` is shaped for
  JSON-stringified `{baseDir}/{key}.json` save slots, a mismatch for that.
  Pulled the actual atomic temp-file-then-rename logic into its own
  exported function, reused by both `createFsStorage.save()` and the
  editor's write endpoint — same "reuse the write mechanics, don't
  reinvent them" principle `editor.md` states, applied at the right level
  of abstraction.
- **Touched-files log needs `git status --untracked-files=all`, not the
  default** — git collapses an entirely-new untracked directory to one
  summary line (`?? src/`) unless told otherwise, which never matches
  provenance recorded per exact file path. Caught during manual
  verification when a written file's label silently failed to show.
- **Same bug class hit twice, in two different files**: Vite's build
  silently stubs any Node builtin import it doesn't recognize as
  explicitly external into an empty, throwing-on-access object — its
  "browser safety net." `storage.js` hit this first (reached via `core`'s
  barrel from any browser build; nothing had ever loaded `core` in a real
  browser context before this session's dev fixture) and was fixed by
  moving `node:fs/promises`/`node:path` into dynamic imports scoped inside
  `createFsStorage`/`writeFileAtomic` themselves. `devServerPlugin.js` hit
  a variant of the same thing purely from the *build step*, not from
  browser loading — it's Node-only dev-server code, never loaded in a
  browser, but still passes through the same Vite build pipeline; fixed by
  adding `node:child_process`/`node:util` to `rollupOptions.external`
  alongside `node:fs/promises`/`node:path`. Caught by inspecting the build
  output directly before it shipped, not by the test suite (which,
  correctly, never loads the built bundle).

## Notes

- Second `/checkpoint-plan`-structured implementation session (after
  session 28): five checkpoints, each stopping for explicit go-ahead after
  a green test run and a commit. Every checkpoint's manual/browser
  verification step surfaced a real bug this session — three of the same
  underlying class (Vite's default Node-builtin browser-stubbing) plus the
  Vite `hot.dispose()` single-slot gotcha and the git untracked-directory
  collapsing behavior. None of these would have been caught by the
  `node --test` suite alone, since it correctly never loads a built bundle
  or a real Vite dev server — this is the first package in the monorepo
  where that gap between "tests pass" and "actually works in a browser"
  was real, and the dev-session skill's manual-verification step earned
  its keep every single checkpoint.
