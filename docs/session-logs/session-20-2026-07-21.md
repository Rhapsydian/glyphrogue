# Session 20 — 2026-07-21

## Goal

Roadmap item 20: AI & behavior, per `docs/design/ai-and-behavior.md` — the
shared shadowcasting/FOV primitive, the shared `findPath` primitive, and
the four first-party `TakeTurn` rules (`Wanders`, `ChasesPlayer`, `Flees`,
`Guards`) with default priorities. The first session that needed actual
entity coordinates: no prior session had instantiated a `Position`
component or given core any way to query a live map.

## Key decisions

One core architectural fork was resolved live with the user before
planning, via `AskUserQuestion`:

**Map-query wiring: injected once at `createApi()`, not threaded as
standalone pure functions.** Core still owns no grid/zone storage
(session 18's decision stands) — `findPath`/`computeFov` both need a
walkability/opacity query over whatever map the game maintains, and the
question was how a first-party, core-shipped `TakeTurn` rule reaches that
query without the game rewriting the rule itself. `isWalkable`/`isOpaque`
callbacks are injected once at `createApi({ ..., isWalkable, isOpaque })`
time, the same DI shape already used for `platform`/`storage`/`rng`; the
rule-dispatch `ctx` (`actions.js`'s `createContext`) closes over them,
exposing `ctx.findPath`/`ctx.computeFov` to every `TakeTurn` rule for
free. The alternative (plain exported pure functions, query passed
explicitly per call) stayed purer in isolation but would have required the
game to manually re-wrap all four first-party rules just to supply the
query, undermining "ship first-party defaults an author can just
register."

Two smaller conventions followed from that decision without needing a
separate ask, since they matched precedent already established elsewhere
in the codebase:

**`Position {x, y}` becomes a real (no longer illustrative-only) core
convention** — it was already `core-architecture.md`'s own ECS example;
this is just the first session that actually needed coordinates.

**First-party behaviors are plain exported functions, not auto-registered
onto `createApi()`** — same precedent as `bspGenerator`/
`cellularAutomataGenerator` etc. ("a generator id is content, not
infrastructure"). A game calls `api.registerRule('wanders', 'TakeTurn',
wandersRule, { priority: WANDERS_PRIORITY })` itself.

## Work completed

- Checkpoint 1: `packages/core/src/fov.js` — `computeFov(origin, radius,
  { isOpaque })`, recursive shadowcasting (8-octant, the classic
  Bergstrom algorithm rot.js's `FOV` module also implements), plus
  `fovContains(fov, x, y)`. Returns a `Set` of `"x,y"`-keyed visible
  cells; an opaque cell is itself visible (it's what blocks the view),
  only what's behind it is excluded. 6 new tests. Committed as `4ecb25c`.
- Checkpoint 2: `packages/core/src/pathfinding.js` — `findPath(from, to,
  { isWalkable, maxNodes? })`, A* over 4-directional adjacency with a
  Manhattan heuristic. Returns ordered steps (exclusive of `from`), `[]`
  if already there, `null` if unreachable. 6 new tests. Committed as
  `967454c`.
- Checkpoint 3: threaded `isWalkable`/`isOpaque` from `createApi()`
  through `createEngine()`'s stored `mapQuery` into `dispatch`/
  `dispatchExclusive`'s `ctx` (`actions.js`), giving every `TakeTurn` rule
  working `ctx.findPath`/`ctx.computeFov`. Also exposed `api.findPath`/
  `api.computeFov` directly, bound to the same injected query, for
  non-rule consumers (rendering's player FOV, light propagation — per
  rendering.md's "one shared primitive, three consumers" framing, only
  one of which goes through the rule pipeline). 6 new tests, including an
  end-to-end one proving a registered rule reaches `ctx.findPath` via
  `api.act()`. Committed as `cc3a2b5`.
- Checkpoint 4: `packages/core/src/behaviors.js` — `wandersRule`,
  `chasesPlayerRule`, `fleesRule`, `guardsRule`, plus their priority
  constants (`FLEES_PRIORITY` 3 > `GUARDS_PRIORITY` 2 >
  `CHASES_PLAYER_PRIORITY` 1 > `WANDERS_PRIORITY` 0, applied by the
  registering caller via `registerRule`'s existing `options.priority`).
  Each gates on its marker component the way the design doc's own example
  does (`if (!ctx.hasComponent(...)) return;`). `ChasesPlayer`/`Flees` use
  `ctx.computeFov` for perception; `ChasesPlayer` then `ctx.findPath`s
  toward the player, while `Flees` greedily steps toward whichever
  walkable neighbor maximizes distance instead (fleeing has no single
  destination for A* to path toward). `Wanders` reuses `ctx.findPath` as
  its own adjacency-walkability check (rather than needing a second raw
  primitive) and deterministically cycles direction, storing
  `lastDirection` on its own component data, since `ctx` has no rng
  dependency to draw true randomness from. `Guards` paths back to a fixed
  `post`. 13 new tests, including the priority tie-break (an actor with
  all four components resolves to `Flees`). Committed as `04da253`.
- Checkpoint 5 (this entry): updated `docs/data-model.md` — `Position`,
  `Wanders`/`ChasesPlayer`/`Flees`/`Guards`, `TakeTurn`, `findPath`,
  `computeFov`, and `createApi`'s `isWalkable`/`isOpaque` injection, all
  moved from "illustrative" to "implemented."
- Full suite: 156/156 `node --test` cases passing (up from 125 at session
  start).

## Deferred / parked

- Exact first-party behavior set beyond the four named here — same
  deferral pattern the design doc already carries (implementation time,
  matching how `mapgen-and-editor.md` deferred exact WFC/CA/BSP
  internals).
- `ChasesPlayer` has no memory of a last-known player position — losing
  sight is an immediate no-op (falling through to `Wanders`), not a
  "search the last-seen spot" behavior. Not required by the design doc;
  would be a natural, separately-scoped enhancement if a game wants it.
- `Guards` has no aggro/attack logic — same scope boundary the design doc
  draws around battle-screen-internal AI; a guard that notices an
  intruder and does something about it beyond returning to post is not
  this session's concern.
- Instantiating a `ZoneDTO`'s blueprint `entities` into live ECS entities
  (with real `Position` components) still isn't built anywhere — every
  test in this session hand-built its own entities/components directly.
  Still "whichever future session first needs a zone actually live,"
  unchanged from session 18's framing; likely session 21 (rendering) or
  the game-runtime layer, not core itself.

## Problem-solving notes

- An early "destination unreachable" `findPath` test used an unbounded
  `isWalkable` (always `true` except a few sealed cells around the goal),
  which sent A* into a genuinely unbounded search over an infinite open
  plane and hung the test process. Real callers always bound `isWalkable`
  to actual zone dimensions (the way `zoneComposition.js`'s
  `isWalkableCell` already does), so the fix was a realistic bounded test
  query, plus a doc comment on `findPath` making that termination
  requirement explicit rather than leaving it an implicit assumption.
- A `computeFov` end-to-end `api.test.js` case similarly hung, for an
  unrelated reason: its `TakeTurn` rule returned no costed follow-on, so
  the scheduler's budget never decreased and `api.run()`'s `while
  (!engine.locked)` loop never reached the `PlayerControlled` actor. Fixed
  by calling `api.act()` directly for a single-turn check instead of
  `api.run()`, which needs at least one actor's budget to actually drain
  to terminate.
- `fleesRule`'s "maximize distance" greedy step turned out to always be
  tied between at least two of the four directions — a mathematical
  property of Manhattan distance under a unit step (moving away on either
  axis increases distance by exactly the same amount), not a bug. Ties
  resolve to whichever direction is listed first; documented in the code
  and accounted for directly in the tests (including the priority
  tie-break test, whose expected flee direction had to be recomputed once
  this was understood) rather than asserting a "uniquely most-away"
  direction that doesn't actually exist for this metric.
