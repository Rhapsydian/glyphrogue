## Key decisions

**Scope gap closed at kickoff.** `registerEntity`/`registerEntityType`
were still marked "doesn't exist yet" in `docs/data-model.md` and absent
from BACKLOG's session-25 description. Given session 25 was flagged as
the last implementation-roadmap session, the user chose to add them here
rather than leave a new deferred item.

**`EventState` ownership (checkpoint 2 fork).** The doc says progress
"lives in an `EventState` component on an entity" without saying which.
Resolved live: each registered event gets its own dedicated tracking
entity, created lazily the first time its trigger matches ã not the
entity that triggered it, since the triggering entity isn't guaranteed to
be the same one a later `waitFor` step concerns (trigger: player enters a
region; `waitFor`: all goblins defeated). Tagged via a `ScriptedEvent`
component rather than a JS-side idãentity map, so progress lives in normal
ECS state and saves/loads for free, per the doc's own claim.

**Timer/scheduler integration (checkpoint 3 fork).** A `timeUnits` wait
needed to plug into `scheduler.js`/`engine.js` without a new engine
primitive. Resolved live: schedule the wait as an ordinary scheduler actor
via the existing `addActor(scheduler, timerEntity, -timeUnits)` ã the
scheduler already tops up every actor's budget by `roundBudget` each round
until someone's budget is non-negative and wins, so a negative initial
budget is a "wait N time units" primitive for free. `engine.js`'s `act()`
gained one more component-tag branch (parallel to the existing
`PlayerControlled` check): a `Timer`-tagged entity dispatches its carried
action instead of `TakeTurn`, then self-removes. Caveat surfaced while
testing: a Timer entity *alone* in the scheduler fires on the very next
`act()` regardless of its negative start ã the wait is only meaningful
relative to other actors also competing for turns, same as any other
budget in this scheduler. Not a bug, just a property of the chosen
mechanism, covered by both `engine.test.js` and `scriptedEvents.test.js`'s
end-to-end case (the latter uses a competing actor to actually exercise
the delay).

**Semver approach (checkpoint 4 fork).** Mod dependency ranges need a
comparator. Resolved live: a hand-rolled `^`/`~`/exact comparator against
`major.minor.patch` only (no pre-release tags, no range combinators),
keeping this project's zero-runtime-dependency posture, over adding the
`semver` npm package (which would have been the project's first runtime
dependency). This project's first version-range check.

**Mod-loading order reuses `registry.js`'s existing generic topological
sort** (applied to mod ids; `core` itself excluded from that graph and
checked separately via `satisfiesRange`) rather than a second sort
implementation ã the same reuse precedent that motivated building
`getOrderedIds` generically back in session 15.

**Recording-api manifest shape (checkpoint 5, discussed with pros/cons).**
Considered grouped-by-kind (`{ rules: [...], generators: [...], ... }`,
matching the doc's own "goblin: 2 rules" framing) against a flat call-order
list. User chose flat, on the reasoning that grouping is trivial to add
later if ever needed, while a pre-grouped structure would permanently lose
true registration order across different kinds.

**Per-mod save-slice wiring**, confirmed briefly rather than a full
stop-and-discuss round (checkpoint 5's other half, lower-stakes per the
plan): extended `save.js`'s existing single `serializeGame`/
`deserializeGame` hook pattern to a list of loaded-mod descriptors, one
independently-versioned slice per mod. A mod's `deserialize` hook receives
the full `{ modDataVersion, payload }` slice and is responsible for its own
migrations, mirroring how `deserializeGame` already receives the raw `game`
slice with no migration step run on its behalf by core.

## Problem-solving notes

- The timer carried-action bug (checkpoint 3) was a same-shape-different-
  key mix-up: `timerCondition()`'s match-condition shape uses `.action` for
  the action-type string (matching `trigger`/`waitFor`'s existing
  convention throughout this module), while a real dispatchable action
  uses `.type` (matching every other action in this codebase). Feeding the
  condition shape directly to `engine.js`'s Timer-dispatch path produced an
  action with `.type === undefined`, which incidentally matched the raw
  scripted-event definition's own registry entry (also `actionType ===
  undefined`, since it has no such field) ã surfacing as `ruleFn is not a
  function` rather than a silent no-op, which is what caught it.
- The api.dispatch() gap (not threading `renderEvents`/`scheduler`) existed
  before this session but had no observable effect until a rule reachable
  only via `api.dispatch()` (a scripted event's trigger) needed
  `ctx.addActor` to actually work. Fixed as part of this checkpoint rather
  than filed as a separate item, since it directly blocked the feature
  being built.
