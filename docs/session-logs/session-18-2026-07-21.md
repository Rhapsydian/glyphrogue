# Session 18 — 2026-07-21

## Goal

Roadmap item 18: build the map-generation **interface and primitives**
designed in `docs/design/mapgen-and-editor.md` — `GenerationContext`,
`registerGenerator`, the composition primitives (stamp template, carve CA,
connect corridor, connectivity pass), the `ZoneDTO` shape, and the
seed+diff save strategy. Actual generation algorithms (BSP/CA/WFC/layered
biome) are session 19's job; this session is the pluggable interface and
primitives those algorithms will call into.

## Key decisions

Three design questions the source doc left open were resolved live with
the user before planning, via `AskUserQuestion`:

**Neighbor-zone lookup is caller-injected, not core-owned.** `core` stays
stateless about zone storage/adjacency — same dependency-injection shape as
`platform`/`storage`/`rng` on `createApi()`. Zone id stays an opaque
caller-defined key; no grid/coordinate system was built in `core` this
session — that's deferred to session 19's *optional* world tier, per
`BACKLOG.md`.

**Logical links are an edge list on the zone (`zone.logicalLinks`), not
entities.** The user asked a sharp follow-up here: does a togglable
teleporter (on/off via a switch) need link-as-entity instead? Resolved by
separating two concerns — the connectivity pass is a one-time topological
check at generation time ("does a path exist"), not a live simulation of
switch/lock state, and the design doc already treats key-gated doors the
same way (locked-until-key is the same shape of problem as off-until-
switch). So the edge list stays the source of truth for whether a link
exists at all; the actual switch/pad a player interacts with at runtime is
just an ordinary entity with its own component/rule, wired to its edge by
sharing the edge's `id`. `runConnectivityPass` trusts a `reachableVia`
stamp's named link once it confirms the link *exists* — it doesn't (and
structurally can't) simulate whether the link is currently toggled on.

**Templates are a `ZoneDTO` plus an `anchors` field**, not a separate DTO
type — matches the design doc's "one zone format" philosophy; `stampTemplate`
treats any `ZoneDTO`-shaped object as stampable material.

One implementation-time deviation from the plan's literal sketch:
`runConnectivityPass` reads links directly off `zone.logicalLinks` rather
than taking a redundant `logicalLinks` option, since the zone already
carries that field.

## Problem-solving notes

- The first version of the "auto-connect prefers a declared anchor over
  nearest-open-cell" test passed for the wrong reason — the anchor and the
  nearest-open-cell fallback were colinear, so both target points produced
  an overlapping carved path and the assertion couldn't actually
  distinguish the two behaviors. Rewrote the test with the anchor and the
  fallback candidate on different columns so the corridor's vertical leg
  provably runs to one and not the other.
- Similarly, the first "reachableVia" test had the link's `from` cell
  already physically reachable via the flood fill on its own, so trusting
  the link vs. not trusting it produced the same outcome. Rewrote it with
  the link's `from` cell embedded in an otherwise-isolated wall pocket, so
  the test only passes if `reachableVia` is genuinely being trusted rather
  than incidentally already-connected.
