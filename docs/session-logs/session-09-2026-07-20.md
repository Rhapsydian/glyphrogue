# Session 09 — 2026-07-20

## Goal

Deep-dive planning session 9 of 9 (the last on the roadmap): **Packaging &
distribution** (`docs/design/packaging.md`). Per `BACKLOG.md`, planning-only,
no engine code — scope was static HTML/GitHub Pages/itch.io build specifics,
Electron + `steamworks.js` integration (achievements, cloud saves),
update/code-signing strategy, and Steam build/page requirements.

## Key decisions

- **Static/Pages/itch deploy mechanics**: `create-glyphrogue-game`'s
  scaffold ships an automated GitHub Pages deploy workflow (credential-free,
  copies `pixelyph`'s `deploy-pages.yml`), but itch.io's `butler push` stays
  a documented manual step — it needs a per-project API key and channel
  name a template can't fill in, matching how `pixelyph` already works
  today.
- **Platform-capability abstraction**: `@glyphrogue/core` gets a thin
  `platform` capability on its existing public API, dependency-injected
  with a no-op default — same shape as `core`'s existing rendering/input
  adapter injection points, not a new architectural pattern. This was
  framed as the session's central fork (build achievements/cloud-save
  awareness into `core`, or leave it entirely to each game's Electron
  layer) and resolved in favor of `core` staying platform-unaware while
  still giving game code one portable API.
- **Achievements**: no generic hook mechanism — purely game-content's job,
  via a rule calling `api.platform.unlockAchievement(id)` as one more
  effect through the existing rule API (`scripting-api.md`'s action/rule
  pipeline already covers arbitrary trigger conditions).
- **Cloud saves**: automatic folder-based Steam Cloud against the
  filesystem save backend `core-architecture.md` already decided (atomic
  temp-file-then-rename writes), not the explicit `ISteamRemoteStorage`
  API. This means cloud saves aren't a `platform.*` method at all — purely
  a Steamworks-dashboard config point. Also settled `core-architecture.md`'s
  deferred "storage backend selection" question: selection stays purely
  per-build-target, injected the same way as `platform`.
- **Electron main/preload IPC bridge**: extends `pixelyph`'s existing
  `electron/main`/`electron/preload` split. Main process initializes
  `steamworks.js` with a try/catch fallback (`steamAvailable` flag) so a
  non-Steam launch degrades gracefully; one IPC handler
  (`platform:unlock-achievement`); preload exposes a single-purpose
  `window.glyphrogueSteam` global rather than a generic grab-bag. The
  Electron scaffold (not `core`) supplies the glue that wires this into
  `core`'s `platform` injection point.
- **Update strategy**: no `electron-updater` in v1. Steam and the itch app
  both self-update installs made through them with zero app code; only
  direct-download lacks an updater, and that gap is accepted and
  documented rather than built around, partly because pairing an
  auto-updater with an unsigned installer would be a worse trust story
  than a one-time manual install.
- **Code-signing**: ships signing-ready but unsigned by default —
  `electron-builder`'s standard env-var-driven signing config is wired in
  unconditionally (builds unsigned if the vars are unset, no error either
  way), so turning on signing later is additive config, not a rewrite.
  Actual certificates/Apple Developer enrollment are inherently
  per-author and out of engine scope.
- **Steam build/page requirements**: split into in-scope technical design
  (one depot per OS platform mapping onto `electron-builder`'s per-platform
  output, `default`/`beta` branch convention, a `steamcmd`+VDF upload
  script template mirroring the itch-`butler` CI pattern, `steam_appid.txt`
  for local dev) versus out-of-scope business/marketing tasks (Steam Direct
  fee, app ID issuance, store page assets, pricing, age ratings) — noted as
  prerequisites handled on Steamworks' own site, not designed here.
