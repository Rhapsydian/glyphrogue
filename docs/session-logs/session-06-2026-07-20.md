## Key decisions

- **Font sources & the shared cell grid**: a tileset can mix font sources
  per-symbol (a web-safe monospace base plus imported glyph sets) rather
  than requiring one merged font — generalizes `rendering.md`'s single-font
  shared-metrics-object decision. Decided the *shape*, deferred the *math*
  (same treatment `rendering.md` gave the exact FOV/shadowcasting
  algorithm): every registered font source carries an explicit, stored
  calibration record (scale factor, baseline offset, horizontal-centering
  mode), computed by a shared default-derivation function but
  override-able and persisted — same "editor tunes live params over the
  runtime's own API" pattern as `mapgen-and-editor.md`.
- **Pixelyph coupling: asset-only boundary**, confirmed — Glyphrogue never
  imports Pixelyph source, only its exported woff/otf + icon-font CSS +
  JSON manifest. The user wanted this made concrete rather than flagged
  vaguely, since they intend to implement it in Pixelyph before Glyphrogue
  implementation starts. Specified four definite manifest/export changes:
  (A) a font-level `meta` block (passthrough of `GlyphSet.js`'s existing
  meta: `pixelsPerEm`, `unitsPerEm`, `baselineRow`, `horizontalPadding`,
  etc.) since these authoring-grid concepts have no standard OpenType
  equivalent a compiled font file can substitute; (B) per-glyph metrics in
  the manifest (name, `advanceWidth`, `offsetX`, width/height — values
  `glyphMetrics()` already computes internally but never surfaces); (C)
  codepoint as the manifest's stable key, since slugs are
  collision-suffixed and can shift across re-exports; (D) manifest
  generation decoupled from the icon-font CSS export specifically, so it's
  produced even when a user exports OTF/WOFF alone. Non-Pixelyph font
  sources don't need this — their own OpenType tables are sufficient.
- **Tileset format**: thin mapping, `symbol -> { fontFace, codepoint,
  paletteToken(s) }` — a font-face + codepoint pair per symbol (not one
  unified font), foreground/background token pair per entry.
  `paletteToken(s)` are **defaults, not exclusive** — confirmed directly
  with the user, who'd asked which was intended: a symbol's tokens render
  by default, but any consumer (a material-tinting rule, mod content, a
  themed reskin) can override with a different token or raw fill at a
  specific placement, same curated-token-plus-raw-color-escape-hatch
  pattern `rendering.md` established for the palette system generally.
- **Material tinting & fill** — resolves `rendering.md`'s one explicitly
  deferred question: does material tinting (recoloring a shared glyph per
  material) need a bitmap-blit tileset rendering backend? No. Tinting is a
  **draw-time fill decision**: canvas sets `ctx.fillStyle` (token color or
  a `CanvasGradient`) before `fillText`; DOM defaults to plain CSS text and
  only switches to inline SVG `<text fill="url(#...)">` when a resolved
  fill is actually a gradient (plain CSS `color` can't gradient-fill text).
  A cell background fill is a second, separate fill under the glyph.
  Pixelyph's unused `backgroundPixels`/`foregroundPixels` glyph sublayers
  were considered and explicitly set aside — not needed, since tinting
  never touches per-glyph pixel data.
