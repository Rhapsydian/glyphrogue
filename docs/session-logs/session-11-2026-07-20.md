## Key decisions

- **Audio is reactive by default, not core-triggered.** Much closer to
  `rendering.md`'s relationship to core than to dialogue's — "play a clang
  on `Attack` connecting" is fully derivable from which action resolved,
  nothing needs to command it. `registerSound(id, { trigger: actionType,
  source, match? })`. A `registerScreen` surface is the one exception,
  calling playback directly for its own private internal moments.
- **No swappable/dependency-injected backend.** Unlike the input adapter
  or the `platform` capability (genuinely platform-divergent), Web Audio
  API is identical across all four declared build targets, Electron
  included (it's Chromium) — one concrete implementation, not an interface
  with exactly one implementation ever behind it.
- Volume/mixing settings fall out for free via the same settings-slice
  mechanism `ui-and-input.md` already established for keybindings.
- Raw Web Audio API vs. a thin wrapper library is genuine implementation-
  time detail, deferred the same way other docs deferred algorithm/library
  choices.
