**UI/UX & input framework** — discussed item-by-item per section, per the
now-established pattern from session 06:

- **Input → action pipeline**: the input adapter lives *outside*
  `packages/core` (raw keyboard/gamepad is a DOM concern; core only
  receives already-resolved input) and maps physical input to **input
  actions** — deliberately distinct from core's own rule-pipeline "Action"
  vocabulary. An **exclusive capture stack** (topmost UI surface claims
  everything, kept simple rather than a declared-per-surface capture set)
  resolves `core-architecture.md`'s open `lock`/`unlock`-vs-DOM-input
  question directly: `unlock()` fires only when a gameplay-intent input
  action reaches the dispatcher with nothing capturing above it.
- **Screen/dialog/menu stack**: UI-owned; core has no concept of it.
  Render and capture are decoupled (every stack entry renders, only the
  top captures). Resolved how a core-side scripted-event step
  (`ShowDialogue`) reaches a stack core doesn't know exists: it sets inert
  core state (e.g. `PendingDialogue`), and the UI layer's subscription
  reacts to that state rather than core calling into UI directly — same
  shape as `rendering.md` exposing FOV/light as queryable results.
- **DOM state binding**: no frontend framework is assumed anywhere in the
  project, so core exposes a framework-agnostic subscribe/notify
  primitive. Notification is **coarse** — one signal per fully-resolved
  core Action (including follow-ons), not fine-grained query
  subscriptions — since Glyphrogue is turn-based, not per-frame, so
  re-reading on every signal is cheap enough not to need dependency
  tracking.
- **Keybinding & remapping**: keyed by input action with an array of
  bindings each (multi-binding support, e.g. arrows + vi-keys at once).
  Default bindings and the input-action vocabulary are game-defined; the
  remap/persist mechanism is reusable. Persists as a settings slice via
  `core-architecture.md`'s storage-backend abstraction, independent of
  world-save data.
- **Controller/gamepad**: feeds the same input-action pipeline as
  keyboard, via internal polling + edge-detection rather than events
  (a Gamepad-API constraint, not a design choice). Discrete press/release
  only — no held-repeat in the adapter, that's a game-level policy.
  Analog stick collapses to a discrete directional input action past a
  deadzone. Keymap bindings gained a `{ device, code/index }` tag to mix
  keyboard and gamepad bindings under one input action. Single active
  gamepad only; local co-op explicitly out of scope.
- **Accessibility**: focus management rides the capture stack (trap/
  restore on push/pop) rather than needing its own mechanism. The canvas
  viewport is an explicit non-goal for screen readers — flagged as a
  future `BACKLOG.md` addon, not designed here. Colorblind support is a
  swappable-palette mechanism only (palettes are already one theme object
  per `rendering.md`); Glyphrogue doesn't ship default color values,
  also flagged as a `BACKLOG.md` addon.
