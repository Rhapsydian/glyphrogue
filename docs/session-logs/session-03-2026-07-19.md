## Key decisions

- **Glyph render mode**: Canvas draws glyphs via `ctx.fillText` against
  the same webfont Pixelyph already exports for DOM UI (rot.js's "font"
  Display mode, not tile mode) — metrics alignment between canvas and DOM
  is automatic by construction, not hand-synced. Viability rests on the
  layered-canvas redraw strategy (see below), not a blind pick. Tileset/
  bitmap blit rendering flagged as a possible v2 backend, not built now.
- **Camera/viewport**: camera state lives in the rendering layer, not
  core/save state — read-only over core's inspection API, matching
  session 1's core/editor boundary. Scroll behavior is **deadzone +
  snap**: camera holds still while the player stays within an inner
  margin, then shifts by exactly enough cells to keep pace when the
  player nears the edge — still integer-cell snap motion, deliberately
  decoupled from the animation system rather than composing with its
  open questions. Smooth pixel-lerp scrolling deferred, gated on the
  animation lock/unlock question being resolved first.
- **Color/palette**: curated token palette (named semantic colors, e.g.
  `danger`, `hp-low`) as the default vocabulary, with a raw-color escape
  hatch for cases a token doesn't fit. One palette/theme object feeds
  both canvas `fillStyle` and DOM CSS custom properties. Per-glyph
  material tinting explicitly out of scope here (fonts-and-tilesets
  session, topic 5).
- **FOV/lighting**: `core` exposes **one shared shadowcasting/visibility
  primitive** (origin + radius + opacity query), reused by three
  consumers — player FOV (feeds render-visibility state), per-monster FOV
  (AI perception, topic 4 territory), and light-source propagation (same
  primitive, graded falloff blended on top). This mirrors session 1's
  "one public API surface" principle instead of implementing FOV
  separately for rendering vs. AI. Remembered (out-of-FOV) cells render
  in a flat memory tone by default, with a theme-level switch to preserve
  last-known light/color instead.
- **Animation**: real-time `requestAnimationFrame` loop decoupled from
  the turn engine's `lock`/`unlock` loop — model state changes instantly,
  rendering tweens between snapshots. Two categories: tweened entity
  movement and transient effects (damage numbers, flashes) that never
  touch model state. Left **fully open, no directional lean**: whether
  `lock()` holds for animation playback duration (serializes turns,
  avoids overlap confusion, risks "AI turn spam" pacing) or allows
  overlap (faster pacing, needs a render-event-buffer architecture) —
  composes with the still-open `act()` contract question from session 1,
  needs its own dedicated research pass.
- **Performance budget**: only the camera viewport is ever drawn, never
  the full map. Layered canvas — static terrain layer redrawn only on
  scroll/map-mutation, entity/effects layer redrawn every animation frame
  but containing only the small set of actually-animating cells. This is
  what makes the `fillText` render-mode decision viable: the perf risk
  was redrawing the whole viewport every frame, not `fillText` itself.
  Scale target for "large map": viewport in the Qud/Cogmind range
  (~80×25 to 100×40 visible cells); underlying map size unbounded.
