## Goal

Not a pre-scoped roadmap topic — the original 9-session roadmap finished

interfaces (a BG3-style skill-check/dice-roll modal, a talent tree, a
custom battle system) into the engine.

## Key decisions

- **`registerScreen(id, {...})`** generalizes `ui-and-input.md`'s one-off
  `ShowDialogue`/`PendingDialogue` pattern into a single `PendingUI({
  screenId, payload })` marker, the same "one interface, many
  implementations" shape as `registerGenerator`.
- **Screen lifecycle**: opening a screen just extends the engine's
  existing per-actor `lock()` for the screen's whole lifetime — no new
  pause primitive needed, since the strictly sequential scheduler already
  suspends everything else for free.
- **Screens are opaque to core**: private internal logic, no rule fires
  for anything happening inside one; closing dispatches exactly one
  ordinary core Action carrying the result, so mods can still hook the net
  outcome without seeing internals.
- **Cosmetic animation delay** needs no new mechanism (same model/view
  decoupling as tweened movement); **interactive mid-resolution** (e.g.
  spending a resource to reroll) is a short `registerScriptedEvent`/
  `waitFor` sequence — no second timing concept.
- **`Attack`/`Damage`/`Death` confirmed a first-party default rule set,
  not an exclusive core mechanism** — same relationship core has to its
  four built-in map generators. A custom battle system defines its own
  action vocabulary and never triggers the default chain.
- Battle systems use the exact same `registerScreen`/pause contract as a
  dice roll — no distinct scheduler needed. Scoped to **turn-based only**;
  real-time-with-pause battle systems are explicitly out of scope.
- Talent trees need no new mechanism — they fall out of ordinary
  entities/components/rules plus one `registerScreen` call.
