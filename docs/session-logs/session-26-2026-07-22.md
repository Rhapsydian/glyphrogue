## Key decisions

- **Framing**: `packages/editor` is an IDE *companion*, not a replacement
  IDE — owns live rendering/interactive tuning and runtime introspection
  (what a code editor structurally can't do), writes outcomes back to real
  project files via a shared dev-server file-write API, everything else
  stays the author's own editor's job.
- **Two new `core` mechanisms**, both motivated entirely by this session's
  tooling needs: `registerRule`'s `components` filter (`all`/`any`/`none`,
  string-or-object entries, 8 comparison operators, partial-match
  semantics, load-bearing — enforced by `dispatch`, not just descriptive)
  resolving rule↔component gating; and `registerGenerator`'s
  `paramsDefaults` (load-bearing defaults `generateZone` actually merges
  in), fixing a real gap where generators declared no params schema at
  all. Both required checking real source (`actions.js`, `bsp.js`,
  `cellularAutomataGenerator.js`) to ground correctly — the generator
  fix in particular needs all four first-party generators migrated to
  exported named constants, not just a docs change.
- **Rules' fuller reads/writes tracking** (beyond gating) resolved via
  declared `{reads, writes}` + an optional dev-mode-only enforcing `ctx`
  wrapper — decided to build now despite no demonstrated UI consumer, the
  one deliberate exception to this session's own "wait for 2+ concrete
  needs" discipline.
- **Shared live-preview rendering primitive**: generalize now (3 real
  consumers — map editor, palette editing, tileset calibration — all
  needing the identical thing). **Narrow shared form primitive**: only
  map editor params + audio mixing actually share the same shape; the
  other three candidate consumers (tileset calibration's enum control,
  keybinding's array-plus-capture-interaction, palette's recursive
  token/gradient structure) stay individually-built. Distinguishing these
  two was the session's clearest lesson in "count consumers isn't enough,
  shapes have to actually match."
- **Terminology split**: Plugin (dev-time, author-managed, in scope) vs.
  Mod (player-facing runtime-toggleable, explicitly out of scope, doesn't
  exist in `core` today) — both previously conflated as "mod."
- **Map editor**: both in-context editing (live world's current zone,
  ephemeral until exported as an override) and standalone authoring
  (scratch zone id, exported as a template or seed+params preset) share
  one `api` instance — no sandboxed second `core` instance needed for
  either. "Static edit mode vs. playtest mode" moved up to be a
  harness-level concept, not map-editor-specific.
- **Content browser** and **tileset/font-calibration editor** are
  documented at first-pass depth only — real foundational decisions
  settled, but each needs its own scoping discussion before an
  implementation session.
