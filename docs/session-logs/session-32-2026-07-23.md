# Session 32 — 2026-07-23

## Goal

Implement `packages/editor` design roadmap item 3, plugin management
(`docs/design/editor.md`'s "Plugin management"/"Services" sections,
`scripting-api.md`'s "Plugin kinds: Content vs. Service"): the combined
core-bundled + author-authored Content list with per-plugin enable/disable,
the folder-per-plugin import/export convention, and a per-slot Services
selector. Unblocked as of session 31 (rename, file-write API, and plugin

## Key decisions

Almost every design question was already resolved in `editor.md`/
`scripting-api.md`; the following were genuine judgment calls this session
had to make on top of that, each grounded in an existing decision rather
than talked through live (this was an implementation session, not a
design-doc session — `.claude/dev-session.md`'s live-decision override
doesn't apply here, same posture sessions 28/29/31 took):

- **Kind (Content vs. Service) is only observable by running a candidate's
  `register()` against `recordingApi.js`** — no plugin object carries a
  static `kind` field. Reused `recordingApi` unmodified for this, exactly
  the tool it already exists for.
- **Directory scan + bootstrap-file parsing happen server-side and return
  already-parsed structured JSON, never raw source.** `devServerPlugin.js`
  has top-level `node:` imports; session 29 already hit a real crash from a
  browser loading Node-builtin imports (`storage.js`). The browser side
  only does the operation that's actually safe there: dynamically importing
  a candidate module and running it against `recordingApi`.
- **New `packages/core` aggregate export, `CORE_PLUGINS`** (`corePlugins.js`)
  — so the editor package doesn't hand-maintain a second copy of the ten
  bundled plugin names, same "derive, don't hand-maintain" posture as the
  mod manifest/`components` filter/touched-files log elsewhere.
- **`dev/main.js` can't double as the scanned bootstrap file** — it also
  mounts the editor and has to keep working regardless. Added
  `dev/sandbox/bootstrap.js` as a stand-in for a real game's own bootstrap.
- **Export is a plain recursive directory copy, not a zip** — `editor.md`
  allows either; picked the copy since it needs no new dependency, matching
  this project's zero-extra-dependency bias elsewhere (`plugins.js`'s
  hand-rolled semver-range subset).
- **Toggling/switching never writes the bootstrap file** — not a judgment
  call, explicit in `editor.md` ("surface a one-line instruction rather
  than silently rewrite a file the tool doesn't fully own"), but worth
  naming because it shapes the whole UI: every action produces a copyable
  instruction snippet, never an immediate mutation.

One real bug found and fixed via manual browser verification (not caught by
the unit tests as first written): a catalog entry's `importName` was always
set to the plugin's true export name, even when the bootstrap didn't
actually import it yet — so the "add this import" line never appeared when
enabling a plugin for the first time. Fixed by only setting `importName`
when the bootstrap already imports it, adding a separate always-present
`suggestedImportName` for the "not yet imported" case, and adding a
regression test that exercises the real `deriveCatalog` →
`buildToggleInstruction` pipeline end to end rather than a hand-built entry
(which is exactly what had let the bug slip through originally).

## Work completed

Six checkpoints, each verified (tests, and browser verification once the
checkpoint 6 fixture existed) before moving to the next:

1. **Discovery & catalog derivation.** `packages/core/src/corePlugins.js`'s
   `CORE_PLUGINS`; `devServerPlugin.js`'s `bootstrapPath` option and
   `/plugins/discover` endpoint (`parseBootstrapSource`'s lightweight
   import/loadPlugins-array text scan, `discoverAuthorPluginFolders`,
   `toModuleUrl` for turning a filesystem path into a browser-importable
   URL relative to Vite's own root); `packages/editor/src/pluginCatalog.js`'s
   browser-safe `deriveCatalog`.
2. **Content list UI.** `PluginList.svelte`, mounted in `App.svelte`;
   `buildToggleInstruction` for the copy-ready bootstrap-edit suggestion.
3. **Services section UI.** `PluginServices.svelte`, per-slot selector;
   `buildServiceSwitchInstruction`.
4. **Import/export.** `/plugins/import` and `/plugins/export` endpoints
   (`isValidPluginId` path-traversal guard, `node:fs/promises`' `cp` for the
   recursive copy); Import field + per-row Export button in `PluginList.svelte`.
   Verified against a real scratch folder: imported, watched it land on disk
   and appear live in the Content list as an author-authored plugin (proving
   the dynamic-import + `recordingApi` pipeline against a real module, not
   just an injected test double); exported via a direct `fetch` call
   (`window.prompt` isn't reliably drivable by the browser automation
   tooling); confirmed duplicate-import and path-traversal rejection.
5. **Error surfacing.** `checkPluginLoadErrors` — a `loadPlugins` dry run
   against `recordingApi`, never a live world, over the catalog's
   `enabledPlugins`; the "Plugin load error" banner in `App.svelte`.
6. **`dev/sandbox` fixture + full end-to-end verification.**
   `dev/sandbox/bootstrap.js` and `dev/sandbox/src/plugins/sample-plugin/`;
   `vite.config.js`'s `bootstrapPath` wiring (required a dev-server restart,
   config isn't hot-reloaded). Confirmed in-browser: `bsp`/`wanders` (core)
   and `sample-plugin` (author) all correctly resolved as enabled against
   the real bootstrap text; disable produced a remove-only instruction;
   switching the `audioLoader` service produced the correct import+add
   instruction; temporarily broke `sample-plugin`'s dependencies to trigger
   a real "depends on ..., which is not registered" error banner, then
   reverted and confirmed it cleared. No console or server errors at any
   point.

`packages/core` test count: 339 → 341 (`CORE_PLUGINS`). `packages/editor`
