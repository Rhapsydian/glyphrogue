## Key decisions

**Deep review** — read the four previously-summary-only docs

`packaging.md`) in full alongside everything already read, cross-checked

**Corrections** (mechanical/quick, batched into one round):
- `packaging.md` no longer claims to be the roadmap's last item (sessions
  10–12 came after it).
- `audio.md`'s sound-asset import resolved via `build-pipeline.md`'s
  existing binary-asset pattern (already solved for fonts) rather than
  left open.
- Renamed `DecideAction` → `TakeTurn` (avoided repeating "Action" in its
  own name, unlike every other action type, and a naming collision with
  `ui-and-input.md`'s deliberately distinct "input action" vocabulary).
- Retracted two overclaimed "this resolves it generally" statements (the
  animation-timing open item; `TakeTurn` multi-rule conflicts) that didn't
  hold up on review.
- Added canvas-access, screen-nesting, and inherited-registration-
  convention clarifications to `custom-ui-and-interactions.md`/`audio.md`.
- Tracked two silently-dropped cross-doc open items in `BACKLOG.md`: the
  zone-diff/overlay format `mapgen-and-editor.md` expected `scripting-
  api.md` to finalize (it never did), and a tileset/calibration editor
  `fonts-and-tilesets.md` expected the UI/UX session to cover (it never
  did either).

**Four substantive resolutions**, worked through one at a time:
1. **`lock()`/`unlock()` only gates real async suspension** (player input,
   an open `registerScreen`) — never animation pacing, since AI-driven
   `TakeTurn` actions resolve synchronously and never call `lock()` at
   all. This confirms the render-event buffer is a **requirement**, not
   the optional "allow overlap" choice `rendering.md` originally posed.
2. **Sound reads off that same shared render-event buffer** instead of
   the coarser per-top-level-action DOM-binding notification — one shared
   ordered-event mechanism serving both consumers, the same "one
   primitive, many consumers" pattern as shared shadowcasting/pathfinding.
   (This resolution itself came from a user follow-up mid-discussion —
   "what if sounds are just entered directly into the render event
   buffer?" — simpler than the notification-payload-trace approach
   originally proposed.)
3. **`TakeTurn` conflicts resolve via explicit priority** — `options.
   priority` as either a constant or a `(action, ctx) => number` function
   resolved fresh at decision time, highest wins, ties fall back to
   dependency-graph load order. A deliberate divergence from the rule
   pipeline's usual additive layering, not an application of it. (Also
   sharpened by a user follow-up: allowing the priority itself to be
   computed dynamically, not just a fixed constant.)
4. **Mid-screen save/reload accepted as a documented v1 gap** —
   player-triggered saves are already blocked by the existing capture
   stack; autosave mid-screen just reopens the `PendingUI` marker fresh
   from its original payload on load, same "accepted gap" treatment as
   `packaging.md`'s `electron-updater`/sandboxing decisions.

**Judgment call: design spec is solid enough for implementation
planning.** A targeted check (not a full re-review) of every doc's "Open
items carried forward" found nothing that blocks starting `packages/core`
— only two candidates (the ECS library bake-off, the `act()` contract),
and both are bounded implementation-time technical tasks, not open design
questions.

**`docs/data-model.md`**: a living reference doc, but standing it up now
as a **skeleton** (section structure + cross-references to source docs,
every entry tagged `illustrative`) rather than a full spec — the field-
level shapes in the design docs are examples, not committed schemas, and
locking them in now would just need immediate revision once real code
lands. Populated during implementation instead, per its own maintenance
rule (a stale entry is a bug in whoever's PR changed the shape, not a
docs-backlog item).

**`packages/core` implementation roadmap**: 12 sessions, dependency-
ordered, sized for ~5-hour token-budget windows, scoped to `packages/core`
only (`editor`/`cli` are later work). Session breakdown planned now;
each session's own internal checkpoints deliberately left to that
session's own plan step, mirroring how the design roadmap itself treated
individual sessions' content.
