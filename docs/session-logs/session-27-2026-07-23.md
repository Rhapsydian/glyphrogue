## Key decisions

- **`packages/editor`'s own UI: Svelte 5, compiled ahead of time, only the
  compiled output published** — replaces the session-26 Preact+htm
  decision. Reasoning: Svelte compiles reactivity away entirely rather
  than diffing a vdom, a better match for this doc's live-tuning surfaces
  (calibration sliders, a params panel reacting to every keystroke).
  Consequence: `packages/editor` becomes the one package in the monorepo
  with a real build step (`exports` points at `dist/`, not `src/`;
  `svelte`/`@sveltejs/vite-plugin-svelte` are devDependencies only, never
  installed by a consuming project).
- **Content browser layout** (previously first-pass only): a facet-able
  list over the manifest (not a fixed hierarchy or separate index-browsing
  UI) for static/registry browsing, mirrored for live/world browsing, plus
  a deliberate cross-navigation shortcut (jump from an entity type to its
  live instances) between the two independently-sourced views.
- **Tileset/font-calibration editor layout** (previously first-pass only):
  two tabs (calibration tuning / symbol authoring), a plain-modal
  confirmation for reference-source changes (no full diff/preview), and a
  non-manifest glyph picker combining raw codepoint input with Unicode
  block-range presets rather than picking one exclusively.
- **Config UI panel layout** — this section had real mechanism decisions
  but, unlike every other tool in the doc, never actually described a
  layout at all. Settled: three tabs (Palette / Keybindings / Audio),
  matching the three-way mechanism split already decided.
- **`paramsDefaults` migration wrinkle resolved**: confirmed (by reading
  the actual generator source, not just presuming) that `wfcGenerator`
  migrates cleanly, but `layeredBiomeGenerator`'s `seedCount` default
  (`biomes.length * 2`) is dynamically computed, not a static value —
  documented as an exception rather than forced into the paramsDefaults
  mechanism, to avoid silently changing behavior for existing callers.
- **Plugin management fully designed** (previously a single thin
  paragraph): derived enable/disable discovery against `loadPlugins`'s
  array, a new folder-per-plugin convention (`src/plugins/<pluginId>/`),
  and author-to-author import/export as a plain filesystem operation —
  prompted by the user asking whether sharing was covered, which it
  wasn't.
- **`mods.js` → Plugin renaming pass pulled forward** from an open-ended
  "someday" deferred item into roadmap item 1, the very next session —
  user's instinct was to do it before any new implementation rather than
  leave it open-ended, since item 1 already touches the same files.
- **Roadmap correction**: `BACKLOG.md`'s roadmap had bundled plugin
  management into "shared UI infrastructure" alongside the live-preview
  and narrow-form primitives, but it doesn't actually depend on either —
  only on the rename (item 1) and the harness's file-write API (item 2).
  Split into its own item 3, renumbering the rest of the list through 9.
