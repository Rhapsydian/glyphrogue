## Key decisions

- **Generation scale**: optional **two-tier** structure — a coarse
  world/region generator producing per-tile biome/adjacency metadata, with
  each tile lazily generating a detailed local zone on first entry. Opt-in,
  not required: a game can use local-zone generation alone, or a single
  hand-authored map, with no world tier. The world tier reuses the same
  generator interface at a coarser scale rather than being a separate
  system.
- **Save strategy**: zones save as **seed/params + a player-mutation diff
  overlay**, not the full generated grid — regenerates deterministically
  from `(world seed, zone id)`, diff reapplies on top. Follows
  `core-architecture.md`'s "persist only essential state" DTO philosophy.
- **Algorithms**: one pluggable **generator interface**
  (`registerGenerator`) with BSP, cellular automata, wave-function-collapse,
  and layered biome generation as built-in first-party strategies, not
  competing architectures — a zone can compose more than one in sequence.
  Game/mod authors can register additional algorithms through the same
  interface.
- **Determinism/seed sharing**: confirmed design goal — players share
  seeds and get the same world. Generators must be pure functions of
  `(seed, params)` with zero hidden entropy. Flags an open
  algorithm-versioning question: does an old shared seed still reproduce
  the same zone after generator code changes?
- **Generator API** (sketched, not full plugin packaging): `registerGenerator(id, generatorFn)`,
  `generatorFn(ctx: GenerationContext) => ZoneDTO`; context provides seeded
  RNG, params, and read access to already-generated neighbor zones.
  Composition primitives (stamp template, carve with CA, connect two
  points, run connectivity pass) so generators compose rather than
  reimplement.
- **Connectivity pass**: mandatory flood-fill reachability check after
  stamping/generation — added directly from the user's follow-up questions
  about whether stamped content could end up isolated, and whether
  teleporters should count as valid connectivity. Reachability modeled as
  a **graph**, not pure grid adjacency: physical edges (walkable
  adjacency) and logical links (teleporters, portals, one-way chutes,
  key-gated doors) both count. Unreachable stamps auto-connect via
  declared template anchor points; `mayBeIsolated`/`reachableVia` let a
  stamp opt out or point at a specific logical link instead.
- **Map editor**: reuses `core-architecture.md`'s existing static edit
  mode — no new editor mode. Params/seed panel + live preview + pin/lock
  (the interactive precursor to overrides). Hand-edited and generated
  zones share one `ZoneDTO` format.
- **Composition**: templates (with declared connection-point anchors),
  stamps (with reachability opt-outs), and overrides — all sharing one
  diff-overlay mechanism with the player-mutation save strategy above.
