## Key decisions

Three architectural forks were resolved live with the user before
implementation, plus one narrower fork mid-checkpoint:

**Font-source calibration default: metrics-based normalization, not
raster measurement.** Every font source (a generic OpenType font's own
tables, and Pixelyph's manifest `meta` block) exposes `unitsPerEm`/
`ascender`/`descender`. Default calibration computes each source's
`verticalSpanRatio = (ascender - descender) / unitsPerEm` and derives
`scale`/`baselineOffset` by comparing it against a reference ratio — pure
arithmetic over metadata every source already publishes, fully
unit-testable with no live ctx/font dependency. Raster-bounding-box
fitting was considered and rejected: it would require measuring actual
rendered glyphs in a live canvas/font context, which can't be unit-tested
the way every other `core` module (`glyphMetrics.js` explicitly, by its own
header comment) has been since session 14.

**Calibration reference is pinnable at registry-creation time, not just
"whichever font source registers first."** Raised by the user mid-session:
a game may know upfront that a Pixelyph-authored icon font is the intended
visual standard, but still want a monospace fallback registered — possibly
in either order, possibly incidentally before the intended standard due to
module load order. `createFontSourceRegistry({ reference: sourceMetrics })`
locks the reference ratio immediately at creation, independent of any
later `registerFontSource` call order; omitting `reference` falls back to
today's simple "first one registered wins" default.

**Cell background fill: two separate primitives (`drawCellBackground`/
`drawGlyphCell`), not one folded function.** Matches the doc's own "second,
separate fill drawn under the glyph" framing and the existing
single-purpose-function discipline. The user explicitly wants this
flexibility preserved for a future optimization — backgrounds change far
less often than the glyph over them, so a later session could decouple
their redraw cadence (terrain layer's dirty-check governing background
redraws, entity layer redrawing glyphs every frame) — now a `BACKLOG.md`
deferred item. `drawTileCell` gives authors one-call convenience anyway
without removing the split, mirroring `paintLayer`'s existing role as a
composition over primitives.

**Palette/font-source/tileset registries stay standalone**, not bound onto
`createApi()` — same precedent session 21 set for `camera`/`glyphMetrics`/
render-layer state: rendering-authoring data with no runtime consumer
inside `core` itself (nothing in `core` dispatches against a tileset or
palette the way `dispatch` calls a registered rule).

**Calibration wiring: `drawGlyphCell` gains optional `offsetX`/
`baselineOffsetPx` overrides**, rather than a bigger signature change —
`tileset.js`'s `resolveSymbol` pre-computes these via
`calibratedGlyphAdvance`/`calibratedBaselineOffset` for a non-base
font-source glyph; every existing single-font-source call site is
unaffected when the overrides are omitted.

**Gradient descriptor shape**: `{ type: 'gradient', stops: [{offset,
color}], direction: 'vertical' | 'horizontal' }` (default `'vertical'`),
mirroring `CanvasGradient.addColorStop`'s own shape directly. A stop's
`color` may itself be a `{ token }` reference, resolved one level — keeps
"tokens are the default vocabulary" consistent even inside a gradient.

## Problem-solving notes

- The first `deriveCalibration`/`baselineOffset` test tripped on floating-
  point imprecision (`0.10000000000000009` vs `0.1`) — switched that one
  assertion to a tolerance check rather than exact equality, same category
  of issue as any floating-point arithmetic test, nothing calibration-
  specific.
- Initially had `terrainDrawCommands`/`entityDrawCommands` always include a
  `background` key (set to `undefined` when absent) — this broke existing
  `assert.deepEqual` tests, since Node's `deepStrictEqual` treats an object
  with `background: undefined` as different from one lacking the key
  entirely. Switched to conditionally spreading the field in only when
  actually present, which is also a more honest "optional field" shape and
  needed no changes to prior tests.
