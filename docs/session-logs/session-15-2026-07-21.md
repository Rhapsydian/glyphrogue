## Key decisions

**Registry scope confirmed live**: `scripting-api.md`'s "dependency-graph
load ordering" describes ordering derived from a *mod's* declared
`dependencies` map, but mod descriptors/loading don't exist until session
25. Built a generic `Registry` now anyway — id uniqueness, self-confirming
`options.override`, plus a per-registration `options.dependsOn: [ids...]`
resolved via topological sort — as a smaller, real mechanism session 25
can later feed resolved mod dependency edges into, rather than inventing
full mod-descriptor semver semantics before there's a mod loader to use
them.

**Dispatch contract designed** (filling a gap the docs leave to
implementation time): a rule handler returns `{ veto: true }`,
`{ followOn: Action[] }`, or nothing. A veto only rejects that specific
action, not already-resolved ancestors/siblings in the same cascade —
matches `core-architecture.md`'s independent-legality chaining
description. Follow-ons process breadth-first (FIFO queue), matching the
"one entry per resolved action, in order" sequencing other docs already
assume.

**Mid-checkpoint design correction**: the first pass at `dependsOn`
validated dependencies eagerly inside `register()`. Testing it surfaced
that this silently forces a dependency to always be registered before
its dependent — making `dependsOn` functionally identical to plain
registration order, which defeats the entire point (tolerating
out-of-order registration, e.g. mod load order not matching dependency
order). Fixed by deferring cycle/missing-dependency validation to
`getOrderedIds()` instead, computed and cached lazily on first call after
any registration — registrations can now happen in any order, including
forward references to not-yet-registered ids.
