## Key decisions

- **Two plugin kinds, not one.** Content plugins (the existing model —
  multi-instance, id-keyed, e.g. the four generators/four behaviors, one
  plugin each) and a new Service plugin kind (single-slot, swappable, for
  `memory`/`audioLoader` — a game has exactly one active implementation of
  a concern at a time, a second doesn't coexist alongside the first, it
  replaces it). See `scripting-api.md`'s new "Plugin kinds: Content vs.
  Service".
- **`api.registerService(id, implementation)`, not raw `api` mutation.**
  The obvious first design (`register(api) { api.ensureMemory = fn }`) was
  checked against `recordingApi.js`, the actual manifest-derivation
  mechanism — it only observes calls to known `registerX` methods, not
  arbitrary property sets, so raw mutation would silently vanish from the
  derived manifest. `registerService` stays consistent with "everything
  meaningful goes through a `registerX`-shaped call": `id` names the
  concern, `implementation` is a bundle of already-bound methods, merged
  flat onto the live `api` by `loadPlugins`. Swapping and dependency
  declarations reuse the existing `override`/topological-sort machinery
  as-is — no new conflict-handling rules needed.
- **Core-bundled plugins ship from `@glyphrogue/core` itself**, not a
  `src/plugins/<id>/` folder — that convention is for author-created
  plugins in a downstream game's own repo. `createApi()` stays
  mechanic-agnostic (no auto-wiring, preserving the existing "a generator
  id is content, not infrastructure" decision); "default-enabled" instead
  means the future `create-glyphrogue-game` scaffold ships them in its
  generated bootstrap by default.
- **Editor discovery needs two sources** (core-bundled vs. author-authored)
  and **Services get a distinct per-slot-selector UI**, not folded into
  content's flat toggle list — a service is single-slot, so "which plugin
  fills `memory` right now" is a different interaction than a checkbox
  list. Both folded into `editor.md`'s existing "Plugin management".
- **Action confirmed not a content kind.** Checked directly against
  `actions.js`: no `registerAction` exists anywhere. An action type is a
  plain string tag, same unregistered shape as Component — corrected an
  early glossary-scoping assumption before it shipped.
- **README's ~260-line session-by-session narrative was pure duplication**
  (mid-close-out redirect) — spot-checked several specific claims against
  `docs/design/rendering.md` and `ai-and-behavior.md` (both already
  document them, in more depth) and confirmed every session number
  referenced has its own `docs/session-logs/` entry. Nothing moved to
  `/docs`; it was already there. Replaced with a short Status section and
  links out.
