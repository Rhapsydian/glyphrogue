## Key decisions

Kickoff research (reading `docs/design/editor.md`'s "Generator composition
tool" section against the actual `packages/core` code) surfaced three gaps
the design doc's prose didn't cover, resolved before planning:

- **`collapseWfc`/`partitionBiomes` need author-declared `tiles`/`biomes`
  data** that no UI in this codebase can produce yet ā€” confirmed not new:
  `wfcPlugin`/`layeredBiomePlugin`'s own registered `paramsDefaults` never
  supply them either, so selecting WFC/layered-biome in the *existing* map
  editor and clicking Generate would already throw. Asked the user how to
  handle it; their call was to ship all 4 generators in the step picker
  with a small, clearly-commented placeholder `tiles`/`biomes` fixture
  baked into `compositionGenerators.js` ā€” explicit testing boilerplate, not
  an authoring surface, not a deferral of the two generators.
- **Two more `packages/core` exports were needed** beyond session 35's
  five: `nearestOpenCell` (for `carveCellularAutomata`'s entry-point
  fallback) and `createZone` (both the live-preview composer and the
  emitted generator's own source need to build a zone from scratch,
  otherwise only reachable via a live `api` instance). Same small,
  deliberate, no-behavior-change precedent session 35 already established
  ā€” made directly, not gated behind live discussion.
- **The design doc's stated emitted signature is stale.** `editor.md` says
  `generatorFn(zone, rng, options)` "matching every other generator's
  existing signature" ā€” but every real generator in this codebase
  (`bspGenerator`, `cellularAutomataGenerator`, `wfcGenerator`,
  `layeredBiomeGenerator`) actually takes a single `ctx` argument
  (`ctx.params`, `ctx.rng`, `ctx.getNeighborZone`), exactly what
  `mapgen.js`'s `generateZone` calls. Implemented against the real, working
  `ctx`-based convention instead of the doc's prose ā€” a factual correction
  grounded in existing code, not an open design question.

## Notes

- The WFC/layered-biome scope question was raised via `AskUserQuestion`
  rather than resolved unilaterally, since it changes the shipped feature's
  scope (not merely an implementation detail) and the design doc gave no
  guidance either way.
- The stale-signature finding (`ctx`-based vs. the doc's
  `(zone, rng, options)`) is the same class of thing session 30 caught
  between `scripting-api.md` and the actual generator/behavior code ā€”
  worth treating design-doc prose about *existing* conventions as a claim
  to verify against the code, not a given, especially when the doc's own
  author (a past session) never actually exercised the convention it
  described.
- **`node --test`'s default file discovery gotcha**: browser-verifying the
  save flow left a real written file at
  `packages/editor/dev/sandbox/src/generators/composed/test-composed-dungeon.js`
  (gitignored, harmless on its own) ā€” but its name matched Node's default
  `test-*.js` test-file glob, so `npm test -w @glyphrogue/editor` silently
  picked it up as a phantom 90th "test" (a zero-assertion pass, named after
  the file path) until it was deleted. Worth avoiding `test-`-prefixed
  names for any manual verification artifact written under `packages/editor`
  in future sessions, or cleaning such artifacts up before the final test
  run rather than after.
