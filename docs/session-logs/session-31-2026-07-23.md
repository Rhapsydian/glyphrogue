## Key decisions

- **`createApi()` restructured to a named `const`.** `registerService`
  needs to `Object.assign` onto the same live `api` object from inside one
  of its own methods, which an anonymous `return { ... }` literal can't
  self-reference. Changed to `const api = { ... }; api.registerService =
  ...; return api;` — no other method's behavior changed.
- **Behavior rules migrate to the `components` filter, not just wrapped
  as-is.** `editor.md` (session 27) already assumes each behavior's rule
  is expressed as `{ all: [marker] }` so the content browser's "which
  rules react to component X" query works mechanically — confirmed by
  re-reading that doc rather than treating it as an open question.
  `behaviors.js`'s four rule bodies had their hand-rolled
  `ctx.hasComponent`/`ctx.getComponent` guard lines removed as part of
  wrapping them, moving the check to registration time instead of leaving
  it duplicated in both places. Three `behaviors.test.js` cases that
  exercised "no-ops without the marker component" were updated to register
  through `registerRule`'s filter rather than relying on the now-removed
  guard.
- **Generator/behavior plugin ids reuse the existing content ids** (e.g.
  `bspPlugin`'s `id: 'bsp'` also registers the generator under `'bsp'`) —
  plugin ids and content ids are separate namespaces per `plugins.js`'s own
  header comment, so no collision risk; reusing the string just keeps
  things readable. Nothing in the docs pinned different values.
- **`memoryPlugin`'s id is pinned to `'memory'`, not arbitrary** —
  `scripting-api.md` requires dependents and override plugins to declare
  `dependencies: { memory: '^1.0.0' }` / `id: 'memory', override:
  'memory'` by that exact name, so the id itself is load-bearing, unlike
  the generator/behavior plugin ids above.
- **No new existing-id/override check inside `registerService`.** Checked
  against `scripting-api.md`'s own framing ("nothing new") — conflict
  handling for services is meant to be enforced at the plugin-level
  dependency/override machinery `loadPlugins` already has, not
  reimplemented as an object-level check.
- **The editor `dev/` fixture's `loadPlugins` call runs unconditionally**,
  not just on a cold start. Plugin registrations (rules/generators/
  services) aren't part of what `serialize`/`deserialize` round-trip —
  only entities/components are — so they need registering fresh every
  time the module runs, restored from an HMR snapshot or not. Also gave
  the fixture's goblin a `Wanders` component so plugin management (next
  session) has something genuinely active to discover and verify against,
  not an inert registry.
