## Key decisions

Two genuinely open forks the design doc left for implementation time,
resolved live with the user before any code was written:

**New package: `packages/input`, not `packages/ui`.** `ui-and-input.md`
decided the input adapter lives outside `packages/core` (core stays a pure
state/rules engine with no DOM dependency), but never named the package â€”
`DESIGN.md`'s original package list still listed "input" under
`packages/core`, which that decision overrode. The broader option
(`packages/ui`, anticipating session 24's screens/audio living there too)
was weighed against a narrower `packages/input` scoped to just this
session. Recommendation was the narrow name: this project's whole
methodology â€” the deep-dive roadmap and every implementation session so
far â€” consistently avoids committing a name/scope to functionality that
doesn't exist yet (the WFC generator stayed minimal, the world/region tier
was deliberately deferred). Naming a package "ui" today for screens/audio
session 24 hasn't designed yet would be exactly that kind of premature
commitment. User asked for pros/cons/recommendation, no override followed.

**Capture-stack scope: a minimal generic stack, not the real screen
stack.** `ui-and-input.md`'s capture-stack decision ("every input action
goes to whichever UI surface is topmost on the stack") is written assuming
the full screen/dialog/menu stack, but that stack (`registerScreen`,
lifecycle, pause contract) is explicitly session 24's job. User picked a
bare push/pop stack of opaque ids â€” used only to gate which input actions
get captured vs. fall through â€” with no rendering, lifecycle, or focus
management. Session 24 builds real screen entries on top of this same
stack.

**`packages/input` stays dependency-free.** Confirmed via reading
`packages/core/src/api.js`/`storage.js`/`index.js` that core's storage
backends are a plain duck-typed `{save, load}` interface, reusable by
`keybindingStorage.js` without an actual `@glyphrogue/core` import â€” keeps
the boundary symmetric (core never reaches into a consumer's domain, and
`packages/input` never reaches into core's package either) and matches
this repo's DI conventions everywhere else (`isWalkable`/`isOpaque`,
`platform`, `storageLike`).

**No game content baked in anywhere in the pipeline.** Which input-action
ids are "gameplay-intent" (â†’ a core Action) vs. UI-only (`confirm`,
`cancel`) is game-defined, same boundary as keybinding defaults and
held-repeat policy â€” `inputPipeline.js`'s `onFallthrough` callback receives
every uncaptured input action and lets the consumer decide, with no
vocabulary opinion baked into this package.
