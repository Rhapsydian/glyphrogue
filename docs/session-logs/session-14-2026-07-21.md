## Key decisions

**ECS library bake-off resolved live, in favor of a purpose-built layer,
not a library.** `core-architecture.md` had named bitECS as a candidate
but flagged a real bake-off at implementation time. Compared bitECS
(typed-array SoA, ~5kb, built-in serialization, but ceremony-heavy —
integer entity IDs, parallel-array component shapes) against miniplex
(plain-object components, no ID bookkeeping, but still an external
dependency to track) — researched both via web search/fetch since actual
current API shape mattered (bitECS has had breaking changes between
0.3/0.4). Landed on a third option instead: purpose-built, ~100 lines,
`Map`-based storage, naive filter query. Reasoning: Glyphrogue is
turn-based (queries run a handful of times per action, not 60x/second),
so a library's real-time performance headroom is mostly wasted; a naive
non-archetype query is trivial at the scale `mapgen-and-editor.md`'s lazy
per-zone generation structurally caps entity counts to (dozens/low
hundreds per zone); and it makes the save/serialization story match
core's own hand-written DTO design exactly, with no library adapter layer
and no third-party API to version against.

**Folder scope confirmed live**: root + `packages/core` only this
session, `packages/editor`/`packages/cli` untouched — matches the
roadmap, per `CLAUDE.md`'s standing instruction to confirm scope before
implementing.
