## Key decisions

- **Module format**: a mod (first-party or, later, end-user) is a single
  default-exported descriptor — `{ id, version, dependencies,
  register(api) }` — one entry point regardless of how the module is
  loaded, rather than named exports per content type (which would grow a
  new export name per content type and doesn't fit function-shaped
  registrations like generators anyway).
- **No hand-authored manifest.** Considered adding a `manifest` export to
  let tooling introspect a mod without executing it, but rejected it for
  drift risk. Instead a manifest is *derived*: dev tooling calls
  `register()` with a **recording** implementation of the same public api
  (logs registrations instead of mutating state) and produces the manifest
  from that — zero drift, one surface for mod authors.
- **Definitions vs. behavior**: entities/items/creatures are inert
  component data (`registerEntity`); behavior is a **rule** keyed to an
  action type and matched by component, not by entity id — so a rule like
  "explode on death" applies to anything tagged `ExplodesOnDeath`, not a
  hardcoded entity list. Multiple rules can register against the same
  action type and run as an **ordered pipeline** (not single-owner), which
  is what lets a mod add reactive behavior without overriding core's own
  rules for that action.
- **`registerEntityType`**: sugar bundling a definition with its own rules
  into one call, decomposing into the same `registerEntity` +
  `registerRule` primitives — not a new runtime concept. Standalone
  `registerRule` stays for genuinely cross-cutting rules not owned by one
  definition.
- **No separate event/hook system.** Hooks are just rules registered
  against action types. Scripted events reuse the same action/rule model;
  `registerScriptedEvent`'s step-list format is declarative sugar for the
  **multi-step case only**, compiling to an `EventState` component + rule,
  which gets save/load and playtest single-stepping for free. `waitFor`
  supports waiting on an action or on elapsed **time-units**, the latter by
  scheduling a timer entity into the same time-units scheduler
  `core-architecture.md` already designed — one mechanism, not two.
- **One consistent registration shape** across `registerEntity`,
  `registerEntityType`, `registerRule`, `registerGenerator`, and
  `registerScriptedEvent`: `id` first, then content args, then optional
  trailing `options`. Conflict handling: hard error on an id collision by
  default; `options.override` must **exactly restate the id being
  registered** (not a `true` flag) so a copy-pasted registration with a
  stale override target fails loudly instead of silently overriding the
  wrong thing.
- **Load order is a dependency graph, not declaration order** — each mod
  declares `dependencies` (including on `core` itself), and load order is
  the topological sort of that graph. This also closes a gap in the
  override design: overriding another mod's registration requires
  depending on that mod, or the override-target-not-registered-yet error
  fires.
- **Save-data integration**: first-party content (statically-imported
  modules) shares one game-owned save slice/version, since it all ships
  and versions with the game; end-user (dynamically-loaded) mods each get
  an **independently-versioned slice**. A save whose mod set isn't fully
  installed **fails to load** (Factorio-style) rather than silently
  dropping or dormant-carrying orphaned mod data — the user explicitly
  preferred this over the two softer options initially proposed, and it's
  consistent with every other ambiguous-state question this session
  resolved as an explicit hard error.
- **Two independent core version numbers**: `coreSchemaVersion` (saves) and
  a separate API version (plugin surface), since the two change on
  different axes. Both mod-to-mod and mod-to-core compatibility reuse the
  same `dependencies` + semver-range mechanism.
- **Sandboxing**: v1 ships none, for first-party or end-user mods —
  explicitly decided, not silently inherited. Documented as a
  platform-dependent risk (a web build still sits inside the browser's own
  tab sandbox; an Electron build has much broader system access). Real
  sandboxing (Worker/iframe + message-passing api) is deferred as a
  prerequisite to actually shipping end-user mod support, not designed at
  implementation level now. Mitigation: authors can disable modding
  **structurally**, per platform or globally, reusing `DESIGN.md`'s
  existing dev/prod build-split pattern (the platform's build entry point
  never imports the mod loader) rather than a runtime flag — one mechanism
  instead of two.
- **Scope boundary**: this doc is the authoring/registration contract only.
  End-user mod discovery/installation/distribution UI is deferred to the
  packaging session (topic 8); constructing the platform-specific build
  entry points the disable mechanism depends on is deferred to the
  build-pipeline session (topic 7).
