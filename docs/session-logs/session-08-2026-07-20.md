## Key decisions

- **Monorepo tooling**: real npm workspaces (`packages/*`), but as a
  Glyphrogue-specific choice justified by needing lockstep pre-1.0
  development — not, as `DESIGN.md` previously implied, something
  `pixelloom`/`pixelyph` already validated. Checking those repos directly
  showed `pixelloom` is a separately-published package `pixelyph` depends
  on normally, with no `workspaces` field and no shared repo. What that
  pair actually validates is the architectural split (small
  dependency-light core + larger app), not the workspaces mechanism.
- **Workspace independence guardrail**: workspace symlinking doesn't
  itself threaten `core`/`editor` being genuinely independent packages
  (versioning and publishing stay separate regardless), but it removes the
  friction that would normally stop `editor` from reaching into
  `packages/core/src/...` by relative path instead of importing
  `@glyphrogue/core` by package name. Decided: `editor`/`cli` only ever
  import `core` by package name, so workspace-resolved dev stays identical
  in shape to what a real external consumer gets. This came out of a
  direct question about the tension between "monorepo" and "independent
  packages" — not something in the original plan.
- **Versioning policy**: independent per-package semver, not lockstep.
  Chosen because it's the option consistent with why `core`/`editor` are
  two packages at all — lockstep would mean a game depending only on
  `core` sees bumps that correspond to nothing it uses. Mechanism (manual
  vs. a tool like Changesets) deferred as implementation detail.
- **Package structure**: two separate scoped npm packages
  (`@glyphrogue/core`, `@glyphrogue/editor`), not subpath exports of one
  package as `DESIGN.md` previously described — resolves a real
  inconsistency, since the packages were already scaffolded (session 1)
  as two independent directories with their own placeholder READMEs.
  Two packages is a strictly stronger structural guarantee (a prod
  bundler never sees `@glyphrogue/editor` in its graph at all, vs. an
  unused `exports` condition it still has to parse).
- **Shipping strategy**: no build step for `core`/`editor` — raw ESM
  `src/`, `sideEffects: false`, `exports` pointing straight at source,
  same discipline `pixelloom` uses (confirmed by reading its actual
  `package.json`: no `dist/`, no bundler script). Sidesteps Vite
  library-mode config and `.d.ts` generation entirely, since the project
  isn't TypeScript anywhere (`pixelyph`/`pixelloom` source is plain
  `.js`/`.jsx`).
- **Asset pipeline scope**: narrowed to fonts only. Maps/mods/config are
  already decided (`scripting-api.md`, `mapgen-and-editor.md`) to be
  ordinary code-imported JS/JSON, not build-time assets — only Pixelyph's
  exported font binaries + manifest are genuine static assets needing
  Vite's normal asset handling.
- **Dev/prod split mechanics**: two HTML entry points in a scaffolded game
  (`index.html` prod, imports only `core`; `dev.html` dev, imports `core`
  + `editor`). `vite build`'s `rollupOptions.input` lists only
  `index.html`, so the prod build has no reachable import path to
  `editor` at all — structural absence, not tree-shaking.
- **`create-glyphrogue-game`**: plain `npm create glyphrogue-game`
  convention, a `templates/default/` directory copied with a few string
  substitutions — no scaffolding framework, matching the project's
  low-ceremony convention.
