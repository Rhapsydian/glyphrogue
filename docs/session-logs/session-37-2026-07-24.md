## Key decisions

Kickoff research (reading `editor.md`'s "Content browser" section against
`packages/core/src/recordingApi.js` and `actions.js`) surfaced one real
gap, same pattern as sessions 35/36 finding a design doc's prose didn't
fully anticipate the real code:

- **`recordingApi.js`'s `registerRule` stub dropped `options` entirely.**
  It only accepted `(id, actionType)`, so no manifest `rule` entry
  anywhere in the codebase ever carried a `components` field — the exact
  data the content browser's "component → rules" cross-reference needs.
  Confirmed via grep that no call site anywhere relied on this data
  existing (the gap was silent, not previously exercised by any test).
  Fixed by extending the stub to destructure and record
  `components`/`reads`/`writes` from a 4th `options` argument, mirroring
  `actions.js`'s own `registerRule` destructure exactly.
- **Static cross-reference indexes use name-only matching, not full
  runtime semantics.** `entityTypeRuleIndex` (entity type → rules that
  would match its instances) is a static, name-only version of
  `actions.js`'s `matchesComponentFilter` — it checks whether a rule's
  `all`/`any`/`none` component names line up against an entity type's
  *declared* component set, but can't evaluate field-level operators
  (`equals`/`gt`/`in`/etc.) since those need a live entity's actual data,
  not just a declared type. A rule with a field predicate can show as
  "would match" a type even though only some real instances actually
  satisfy it — the live view is where that distinction becomes checkable.
  Documented as a deliberate simplification in `contentCatalog.js`, not
  raised as an open question, since there's no live data to check
  against at the static/type level.
- **The "show live instances" cross-nav needs to require *every* declared
  component, not just one** — caught during browser verification, not
  planning. The first implementation pre-filtered the live view using
  only the entity type's first declared component (`entry.components[0]`)
  via the existing single-select dropdown, which under-filtered (an
  entity missing some of the type's other components still matched).
  Fixed by adding a separate `requiredComponents` AND-filter (distinct
  from the manual single-component dropdown), with its own
  breadcrumb/Clear affordance mirroring the registry view's existing
  cross-reference breadcrumb pattern.
