## Goal

engine code.

## Key decisions

- **Data model**: entities/components (ECS-style, candidate library
  `bitECS`) for data storage, but game logic as **actions + rules**
  instead of per-frame systems — plain ECS systems don't sequence well for
  turn-based play.
- **Scheduling**: a **time-units** scheduler (variable-cost energy-budget
  model, à la Cogmind's time-energy system, but explicitly not using that
  label — "time units" instead), driven by a rot.js-style async engine
  loop with `lock`/`unlock` so a turn can suspend cleanly on player input.
- **Save/load**: DTO pattern, not raw ECS serialization. Two independently
  versioned slices per save — `coreSchemaVersion` (core's own format,
  core owns migrations) and `gameDataVersion` (downstream game/mod data,
  game owns migrations) — using a shared versioned-migration mechanism.
  JSON format, storage backend abstracted (localStorage vs. filesystem
  decided later, in the packaging session).
- **`core`/`editor` boundary**: `core` exposes one public
  inspection/mutation API; the game runtime, `editor`, first-party
  content, and end-user mods are all consumers of the same surface — no
  privileged backdoor for `editor`. Sets up `editor` to potentially ship
  as a standalone "mod tools" release later without redesigning the
  boundary (distribution question itself deferred to the packaging
  session).
- **Modding scope clarified**: two distinct things, both real value —
  (1) a game's own content authored via the plugin API (core
  infrastructure, needed for any game), and (2) end-user mods for a
  shipped game (confirmed as wanted, but distribution/compatibility/
  sandboxing deferred to the scripting-api and packaging sessions).
  Flagged that `DESIGN.md`'s existing "no sandboxing in v1" decision needs
  revisiting specifically for end-user mods, not just first-party content.
- **Editor preview**: two modes rather than one — **static edit mode**
  (scheduler off, direct edits via the inspection/mutation API) and
  **playtest mode** (real scheduler running, with both continuous-run and
  single-step controls so a developer can step through scripted events one
  action at a time).
