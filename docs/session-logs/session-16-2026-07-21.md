## Key decisions

**Priority/exclusive resolution designed as a second dispatch mode, not a
`TakeTurn`-specific special case.** Session 15's `dispatch()` is additive
(every matching rule runs, all `followOn`s collected) — right for
`Attack`/`Death`, wrong for `TakeTurn` (`ai-and-behavior.md`: mutually
exclusive, an actor can't both `Flee` and `Guard` the same turn). Added
`dispatchExclusive(world, registry, action)`: runs every matching rule,
keeps only the highest-`options.priority` result (constant or
`(action, ctx) => number`), discards the rest with no partial
application. Ties keep the earlier-registered rule, per the "ties fall
back to load order" decision. The engine calls this specifically for
`TakeTurn`; ordinary game code dispatching `Attack`/`Move` still uses
plain `dispatch()`.

**Action cost travels on the action object itself**
(`{ type: 'Move', cost: 100, ... }`), matching "a heavy attack costs more
time than a step" (variable per-instance cost, not a fixed per-type
table). The engine sums `cost ?? 0` across every action a turn resolves
to and spends the total from the scheduler.

**Player vs. non-player actors distinguished by a new `PlayerControlled`
marker component**, reusing the existing component system rather than
inventing a new engine-level concept. `act(engine)` locks and waits on a
`PlayerControlled` actor's turn without dispatching anything; external
code is expected to call the new `resolvePlayerAction()` once the
player's action is chosen. Both `PlayerControlled` and the `cost` field
were added to `docs/data-model.md`, sourced to the actual implementation
file rather than a design doc, per that doc's own maintenance rule.

**Scheduler is a fixed-per-round energy budget**: every actor holds a
budget; `next()` returns the highest-budget actor, self-refilling
everyone by a fixed `roundBudget` first if the current max is `<= 0` —
keeps `next()` self-sufficient rather than requiring the engine to manage
round boundaries separately. Budgets can go negative (front-loading an
expensive action), matching the time-units decision directly.

**Explicitly out of scope**: the real `Wanders`/`ChasesPlayer`/`Flees`/
`Guards` behavior *content* — needs `ctx.findPath` and the shared
shadowcasting/FOV primitive, both scoped to session 20. This session
built and tested the mechanism only, using throwaway test rules.
