# Session 17 — 2026-07-21

## Goal

Roadmap item 17: formalize `packages/core`'s public inspection/mutation API
surface (sessions 14-16 built the ECS/registry/actions/scheduler/engine
layer as free functions taking `world`/`registry`/`scheduler` as explicit
arguments), plus save DTOs with a versioned migration mechanism, a storage
backend abstraction, the no-op-default `platform` capability, and headless/
deterministic testability — the piece game authors need before they can
write their own `node --test` files directly against this API.

## Key decisions

**Public API is a bound game-instance object, not a re-export of the
existing free functions.** `core-architecture.md`/`scripting-api.md`'s own
examples (`api.registerRule(id, ...)`, `ctx.hasComponent(entity, type)`)
show no manual world/registry threading — resolved with the user before
planning that `createApi()` should close over one internal
world+registry+scheduler+engine instance and return bound methods, the same
pattern `actions.js`'s internal `createContext(world)` already used for
rules, just promoted to the public surface (`packages/core/src/api.js`).

**Session scoped as one pass through all five roadmap sub-pieces**, run as
checkpoints with a pause-and-report after each commit, rather than
splitting part of it to a follow-up session — also resolved with the user
up front, since the roadmap explicitly allows re-splitting once a session
is scoped against real code (same as sessions 18/19 did for mapgen).

**`platform`/`rng` are dependency-injection points on `createApi()`**, same
shape as the existing storage injection `packaging.md` already specified —
`platform` defaults to a harmless no-op (`unlockAchievement`), `rng` is a
small zero-dependency seeded PRNG (mulberry32) whose `state` is a plain
number so save/load can snapshot/restore it directly with no separate
getter/setter pair.

**Registry state is deliberately not part of the save DTO.** Rules/
generators/etc. are code, reconstructed by re-running a game's own
`register*` calls at boot — only world/scheduler/rng *state* persists. This
is why `deserialize()` returns a fresh, empty-registry `api` instance; a
real load sequence re-registers content first, then loads the save on top,
exactly as `headless.test.js` demonstrates.

**Migrations are a small sparse-chain runner** (`runMigrations(payload,
fromVersion, toVersion, migrations)`, keyed by *target* version so gaps
don't need placeholder steps), shared by the `core` slice now and available
for `game`/mod slices once those exist — no schema changes have actually
happened yet, so this session's only migration content is a synthetic test
fixture, not a real historical migration.

**A save's mod slice must exactly match the currently-installed mod set**,
per `scripting-api.md`'s decision — `deserialize()` hard-errors if `dto.mods`
references an id not in `registeredModIds`. Nothing produces a real mod
slice yet (no mod-loading system exists), so this is exercised with a
synthetic fake mod id in tests, not real mod content.

**Storage backend abstraction needed no Electron dependency** — `packages/
editor`/`packages/cli` are both still empty placeholders, and Electron
isn't wired into this repo at all yet, but `createFsStorage`'s atomic
temp-file-then-rename writes are plain `node:fs/promises` code that
Electron's main process would use unchanged. `createLocalStorageBackend`
takes its storage object as a parameter instead of reading
`globalThis.localStorage` directly, so it's fully testable under
`node --test` with a small fake `{getItem, setItem}` object, no DOM needed.
