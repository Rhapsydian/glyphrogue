# Session 01 ‚Äî 2026-07-19

## Goal

Kick off the Glyphrogue project: a web-based ASCII/glyph roguelike game
engine inspired by Caves of Qud. Scope for this session was explicitly
research and planning only ‚Äî no engine code.

## Key decisions

- **Name: Glyphrogue** (Glyph + roguelike). Went through three rounds of
  naming ‚Äî Glyphscii (rejected, "scii" swallows the A from ASCII and reads
  awkwardly), a round of terminal/console-themed options (Glyphterm,
  Glyphshell, Glyphansi, Glyphcursor), before landing on Glyphrogue.
- **Repo structure**: npm-workspaces monorepo (`packages/core`,
  `packages/editor`, `packages/cli`), mirroring the pattern already
  validated in `pixelloom` ‚Üí `pixelyph` ‚Äî a small, dependency-light core
  library wrapped by a larger app with UI/tooling.
- **Rendering: hybrid Canvas 2D + DOM/CSS grid.** Canvas for the game
  viewport (map/entity glyph grid, same approach as rot.js), DOM for menus/
  HUD/dialogs/inventory ‚Äî per explicit user request, both sharing one
  glyph-metrics source.
- **Scripting: plain TS/JS modules** against a plugin API, no sandboxing
  in v1.
- **Desktop/Steam target: Electron, not Tauri.** Researched specifically
  because generic Tauri-vs-Electron comparisons don't address Steamworks
  SDK support ‚Äî found that `steamworks.js` and equivalent wrapper libraries
  only officially support Electron/NW.js, and Tauri devs report real
  friction getting Steamworks working on Steam Linux. Also matches tooling
  already proven in `pixelyph` (`electron-vite` + `electron-builder`).
- **Map generation is a core runtime feature**, not a dev-only tool ‚Äî it
  ships in every game. The dev-time map editor is a design-time surface
  over the same procgen API a shipped game calls at runtime, not a
  separate map format.
- **Scope**: user wants an extensive, multi-session deep-dive planning
  pass before any implementation starts. Broke this into an 8-topic
  roadmap in `BACKLOG.md`, roughly in dependency order (core architecture
  and rendering first, packaging last), each producing its own doc under
  `docs/design/`.
